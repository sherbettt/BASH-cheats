### Симптомы и обнаружение проблемы

Обнаружили, что сервис `runtel-event-sender-v2.service` не работает. При проверке статуса командой:

```bash
systemctl status runtel-event-sender-v2.service -l --no-pager
```

Было получено сообщение о падении сервиса:

```
Active: failed (Result: exit-code)
Process: 110923 ExecStart=/opt/runtel/bin/event_sender -c /etc/runtel/event-sender.yaml (code=exited, status=1/FAILURE)
```

### Почему это критично

Сервис `runtel-event-sender-v2` отвечает за отправку событий от FreeSWITCH (медиа-сервера) в системы мониторинга и биллинга. Без него:
- Не обрабатываются события звонков (регистрация, ответ, завершение)
- Не работают verto-соединения
- Не отслеживается состояние каналов и конференций
- Нарушается работа callcenter-функционала

**Проблема требовала немедленного решения.**

---

## 🔧 Диагностика: как искали причину

### Шаг 1. Анализ логов сервиса

```bash
journalctl -xeu runtel-event-sender-v2
```

**Результат:** `-- No entries --` — логи отсутствовали, что затрудняло диагностику.

### Шаг 2. Ручной запуск программы

Для получения реальной ошибки выполнили:

```bash
/opt/runtel/bin/event_sender -c /etc/runtel/event-sender.yaml
```

**Получили ошибку:**
```
FileNotFoundError: [Errno 2] No such file or directory: '/etc/runtel/event-sender.yaml'
```

**Вывод:** Конфигурационный файл отсутствовал.

### Шаг 3. Создание конфигурационного файла

```bash
mkdir -p /etc/runtel
mcedit /etc/runtel/event-sender.yaml
```

### Шаг 4. Обнаружение ошибки в синтаксисе YAML

При повторном ручном запуске получили новую ошибку:

```python
schematics.exceptions.DataError: {"hosts": {"2": ["Invalid IPv4 address"]}}
```

**Причина:** В секции `cache.hosts` был неправильный синтаксис — два IP-адреса на одной строке:

```yaml
hosts:
  - 172.16.15.36
  - 172.16.15.37
  - 172.16.15.38
    10.172.108.85    # ← ошибка!
```

**Исправление:** Каждый IP на отдельной строке.

### Шаг 5. Новая ошибка — подключение к FreeSWITCH

После исправления YAML программа запустилась, но появилась ошибка подключения:

```
WARNING text_rude_rejection Access Denied, go away
```

**Что это значит:** FreeSWITCH на сервере `172.16.15.36` отклонял подключение по ACL (Access Control List).

### Шаг 6. Диагностика подключения к FreeSWITCH

```bash
telnet 172.16.15.36 8021
```

**Ответ:** `Access Denied, go away` — подключение блокируется.

### Шаг 7. Выявление архитектурной проблемы

В ходе диагностики выяснилось, что у нас **два разных сервера с FreeSWITCH**:

| Сервер | IP | Роль |
|--------|-----|------|
| `a43` | `172.16.15.76` | Локальный сервер, где запущен `event_sender` |
| `msk-vps1` | `172.16.15.36` | Удалённый сервер, куда пытался подключиться `event_sender` |

**Проблема:** Конфиг `event_sender` был настроен на подключение к удалённому серверу `172.16.15.36`, а не к локальному FreeSWITCH.

---

## 🛠️ Решение: что меняли и в каких файлах

### Изменение №1. Создан конфигурационный файл

**Файл:** `/etc/runtel/event-sender.yaml` (сервер `a43`)

**Действие:** Файл отсутствовал → создан с нуля.

---

### Изменение №2. Исправлен синтаксис YAML

**Файл:** `/etc/runtel/event-sender.yaml` (сервер `a43`)

**Было (неправильно):**
```yaml
cache:
  hosts:
    - 172.16.15.36
    - 172.16.15.37
    - 172.16.15.38
      10.172.108.85    # ← два IP в одной строке
```

**Стало (правильно):**
```yaml
cache:
  hosts:
    - 172.16.15.36
    - 172.16.15.37
    - 172.16.15.38
    - 10.172.108.85    # ← отдельный элемент списка
```

---

### Изменение №3. Перенаправление на локальный FreeSWITCH

**Файл:** `/etc/runtel/event-sender.yaml` (сервер `a43`), секция `media_server`

**Было:**
```yaml
media_server:
  password: RuntelSecter2020
  host: 172.16.15.36      # ← удалённый сервер
  port: 8021
```

**Стало:**
```yaml
media_server:
  password: RuntelSecter2020
  host: 172.16.15.76      # ← локальный сервер a43
  port: 8021
```

**Почему:** Локальный FreeSWITCH уже настроен и работает, не требует правки ACL.

---

### Изменение №4. Проверка локального FreeSWITCH

**Файл:** `/etc/freeswitch/autoload_configs/event_socket.conf.xml` (сервер `a43`)

**Результат проверки:** Конфиг корректен, изменения не потребовались:

```xml
<configuration name="event_socket.conf" description="Socket Client">
  <settings>
    <param name="nat-map" value="false"/>
    <param name="listen-ip" value="172.16.15.76"/>
    <param name="listen-port" value="8021"/>
    <param name="password" value="RuntelSecter2020"/>
    <param name="apply-inbound-acl" value="acl3"/>
  </settings>
</configuration>
```

**Почему не меняли:** Пароль совпадает, порт открыт, ACL не блокирует локальные подключения.

---

### Изменение №5. (Опционально) Планировалось на msk-vps1, но не потребовалось

**Файл:** `/etc/freeswitch/autoload_configs/event_socket.conf.xml` (сервер `msk-vps1`)

**Планировалось:**
```xml
<!-- <param name="apply-inbound-acl" value="acl3"/> -->
```

**Итог:** Изменение не потребовалось, так как сервис больше не обращается к этому серверу.

## 🎯 Ключевые выводы

1. **Важно проверять наличие конфигурационных файлов** — их отсутствие приводит к неочевидным ошибкам
2. **Синтаксис YAML критичен** — даже один лишний пробел или неправильный отступ ломает парсинг
3. **Диагностика через ручной запуск программы** — самый эффективный способ увидеть реальную ошибку
4. **Проблемы с FreeSWITCH ACL** часто решаются проще перенастройкой целевого сервера, чем правкой ACL
5. **Локальный FreeSWITCH предпочтительнее удалённого** — меньше проблем с сетью и ACL
