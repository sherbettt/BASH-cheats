
## 1. Скрипт `/usr/lib/cgi-bin/cti-panel` с комментариями

```bash
#!/bin/bash
# Указываем интерпретатор — bash

# === ЗАГОЛОВКИ HTTP (обязательно для CGI) ===
echo "Content-Type: text/html"
# Говорим браузеру, что отдаём HTML-страницу

echo ""
# Пустая строка — разделитель между заголовками и телом ответа

# === ГЕНЕРАЦИЯ HTML ===
# Запрашиваем список файлов из GitLab API
curl -s -H "PRIVATE-TOKEN: glpat-uyy-DXxy2iXh5eFZxWyB" \
  "https://gitlab.runtel.org/api/v4/projects/86/packages/310/package_files" \
  | jq -r '.[] | "<a href=\"/cti/download/" + (.id|tostring) + "\">" + .file_name + "</a><br>"'
# - curl -s: тихий режим (без прогресс-бара)
# - -H "PRIVATE-TOKEN: ...": заголовок с токеном для авторизации
# - API-эндпоинт: возвращает JSON со списком файлов в пакете 310
# - jq -r: обрабатывает JSON и выводит сырые строки (без кавычек)
#   .[] — перебирает каждый элемент массива
#   "<a href=\"/cti/download/" + (.id|tostring) + "\">" + .file_name + "</a><br>"
#   — формирует ссылку для каждого файла:
#     * /cti/download/ID — куда будет вести ссылка
#     * .file_name — отображаемое имя файла
#     * <br> — перевод строки (для простого списка)
```

## 2. Конфиг Nginx `/etc/nginx/sites-enabled/ext/registry.runtel.ru` с комментариями

```nginx
# === БЛОК 1: HTTP → HTTPS редирект ===
server {
    server_name registry.runtel.org;   # Домен, на который реагируем
    listen 80;                         # Слушаем порт 80 (HTTP)
    return 301 https://registry.runtel.org$request_uri;
    # Перенаправляем на тот же URL, но по HTTPS (код 301 — постоянный редирект)
}

# === БЛОК 2: ОСНОВНОЙ HTTPS-СЕРВЕР ===
server {
    server_name registry.runtel.org;   # Домен
    listen 443 ssl;                    # Слушаем порт 443 (HTTPS)
    charset utf-8;                     # Кодировка
    client_max_body_size 5M;           # Максимальный размер тела запроса (для загрузки)
    
    # Пути к логам
    access_log /var/log/nginx/registry.runtel.org-access.log;
    error_log /var/log/nginx/registry.runtel.org-error.log;
    
    # SSL-сертификаты
    ssl_certificate /etc/nginx/ssl/runtelorg.pem;
    ssl_certificate_key /etc/nginx/ssl/runtelorg.pem;

    # === БЛОК 3: API (возвращает JSON) ===
    location /api/cti/ {
        limit_except GET HEAD { deny all; }
        # Разрешаем только GET и HEAD (запрещаем POST, PUT, DELETE)

        proxy_pass https://192.168.87.10/api/v4/projects/86/packages/310/package_files;
        # Проксируем на внутренний GitLab API (возвращает JSON)

        proxy_set_header Authorization "Bearer glpat-uyy-DXxy2iXh5eFZxWyB";
        # Добавляем токен для авторизации в GitLab

        proxy_set_header Host gitlab.runtel.org;
        # Подменяем Host, чтобы GitLab "думал", что запрос пришёл на его имя

        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        # Передаём реальный IP и протокол клиента

        proxy_http_version 1.1;
        # Используем HTTP/1.1 для keep-alive
    }

    # === БЛОК 4: CGI-СКРИПТ (генерирует HTML) ===
    location /cti/ {
        include fastcgi_params;
        # Подключаем стандартные параметры FastCGI (переменные окружения)

        fastcgi_pass unix:/var/run/fcgiwrap.socket;
        # Передаём запрос в fcgiwrap через Unix-сокет

        fastcgi_param SCRIPT_FILENAME /usr/lib/cgi-bin/cti-panel;
        # Указываем путь к CGI-скрипту, который нужно выполнить
    }

    # === БЛОК 5: РЕДИРЕКТ /cti → /cti/ ===
    location = /cti {
        return 301 /cti/;
        # Если пользователь заходит на /cti (без слеша), перенаправляем на /cti/
    }

    # === БЛОК 6: СКАЧИВАНИЕ ФАЙЛОВ ===
    location ~ ^/cti/download/(\d+)$ {
        limit_except GET HEAD { deny all; }
        # Разрешаем только GET и HEAD

        proxy_pass https://192.168.87.10/api/v4/projects/86/packages/310/package_files/$1/download;
        # Проксируем на GitLab API для скачивания
        # $1 — это ID файла, захваченный из URL

        proxy_set_header Authorization "Bearer glpat-uyy-DXxy2iXh5eFZxWyB";
        proxy_set_header Host gitlab.runtel.org;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_http_version 1.1;
        # Те же заголовки, что и в API-блоке
    }

    # === БЛОК 7: ВСЁ ОСТАЛЬНОЕ — 404 ===
    location / {
        return 404;
        # Любой запрос, не попавший ни в один из блоков выше, получает 404
    }
}
```

## Отдельное объяснение работы

### Как работает скрипт `cti-panel`

1. **Вывод заголовков:** `echo "Content-Type: text/html"` и пустая строка — это обязательные заголовки для CGI, чтобы браузер понял, что получает HTML.
2. **Запрос к API:** `curl` с токеном обращается к GitLab API, получает JSON-список файлов.
3. **Генерация HTML:** `jq` преобразует JSON в HTML-ссылки вида:
   ```html
   <a href="/cti/download/508">CTI-panel-v2.3.0.620.msi</a><br>
   ```
4. **Результат:** Браузер получает простой список ссылок на скачивание.

### Как работают блоки Nginx

| URL | Что происходит | Что получает клиент |
|-----|----------------|---------------------|
| `https://registry.runtel.org/cti/` | Nginx запускает CGI-скрипт `/usr/lib/cgi-bin/cti-panel` | HTML-страница со списком ссылок |
| `https://registry.runtel.org/cti/download/509` | Nginx проксирует запрос на GitLab API: `/api/v4/projects/86/packages/310/package_files/509/download` | Файл (ZIP-архив, DEB, RPM, MSI) |
| `https://registry.runtel.org/api/cti/` | Nginx проксирует запрос на GitLab API: `/api/v4/projects/86/packages/310/package_files` | JSON-список файлов (для разработчиков) |
| Любой другой URL | Nginx возвращает 404 | Страница "Not Found" |

### Как это всё связано

```
Клиент → https://registry.runtel.org/cti/
              ↓
         Nginx (front2)
              ↓
    запускает CGI-скрипт (/usr/lib/cgi-bin/cti-panel)
              ↓
    скрипт через curl запрашивает GitLab API
              ↓
    получает JSON, парсит через jq, генерирует HTML
              ↓
    возвращает HTML-страницу клиенту
              ↓
    Клиент видит ссылки, кликает на /cti/download/509
              ↓
         Nginx проксирует запрос на GitLab API
              ↓
    GitLab возвращает файл
              ↓
    Nginx отдаёт файл клиенту
```

## Полная инструкция проделанной работы

### Цель
Настроить публичный доступ к пакетам CTI-panel через домен `https://registry.runtel.org`, чтобы клиенты могли видеть список файлов и скачивать их.

### Шаг 1: Создание CGI-скрипта
**Файл:** `/usr/lib/cgi-bin/cti-panel`

Содержимое:
```bash
#!/bin/bash
echo "Content-Type: text/html"
echo ""

curl -s -H "PRIVATE-TOKEN: ваш_токен" \
  "https://gitlab.runtel.org/api/v4/projects/86/packages/310/package_files" \
  | jq -r '.[] | "<a href=\"/cti/download/" + (.id|tostring) + "\">" + .file_name + "</a><br>"'
```

**Права:**
```bash
chown www-data:www-data /usr/lib/cgi-bin/cti-panel
chmod 755 /usr/lib/cgi-bin/cti-panel
```

### Шаг 2: Настройка Nginx
**Файл:** `/etc/nginx/sites-enabled/ext/registry.runtel.ru`

Скопируйте конфиг с комментариями (см. выше).

**Проверка и перезагрузка:**
```bash
nginx -t && systemctl reload nginx
```

### Шаг 3: Установка и запуск fcgiwrap
```bash
apt install fcgiwrap
systemctl enable fcgiwrap
systemctl start fcgiwrap
```
