# диагностика и решении проблемы с переполнением диска логами

## 1. Исходная проблема

**Симптомы:**
- Диск `/` (74 ГБ) был заполнен на 42,8 ГБ из-за огромного файла `/var/log/syslog.1` размером **22 ГБ**
- Система работала медленно, возможно, были ошибки записи

**Первоначальные команды (ncdu):**
```bash
ncdu /          # показал, что /var занимает 42,8 ГБ
ncdu /var       # показал, что /var/log занимает 31,8 ГБ
ncdu /var/log   # показал, что syslog.1 = 22 ГБ, syslog = 8 ГБ
```

---

## 2. Что мы сделали и зачем

### 2.1. Очистка логов (немедленное освобождение места)

```bash
sudo truncate -s 0 /var/log/syslog.1   # очистили огромный файл (было 22 ГБ)
sudo truncate -s 0 /var/log/syslog     # очистили текущий лог (было 8 ГБ)
sudo systemctl restart rsyslog         # перезапустили службу логирования
sudo journalctl --vacuum-size=200M     # ограничили журнал systemd до 200 МБ
```

**Зачем:**
- Немедленно освободить место на диске
- Предотвратить дальнейший рост логов до выяснения причины

**Результат:**
- Освободили **~30 ГБ** (22 + 8)
- Диск стал: 31 ГБ использовано из 74 ГБ (37% занято)

---

### 2.2. Диагностика причины аномального роста логов

```bash
sudo tail -f /var/log/syslog   # просмотр "живых" сообщений
```

**Что нашли:**
- Тысячи повторяющихся ошибок PostgreSQL вида:
  ```
  ОШИБКА: отношение "30120__dialer_attempt" не существует
  ОШИБКА: отношение "372__dpe_result" не существует
  ```
- Ошибки генерировались процессом `patroni` (менеджер кластера PostgreSQL)
- Скорость: **сотни ошибок в секунду**

---

### 2.3. Поиск источника ошибок

```bash
ps aux | grep patroni                    # нашли процессы Patroni
sudo lsof -i :5432                       # посмотрели, кто подключён к PostgreSQL
ps aux | grep -E "2048815|2048827"       # детали конкретных процессов
sudo cat /proc/2048815/cmdline           # узнали, что за программа
```

**Что нашли:**
- Виновник: **`runtel_pbx_v2_master`** — система телефонии/биллинга
- Процессы:
  ```
  /home/billing/runtel_pbx_v2_master/venv/bin/python3 iface.py
  ```
- Приложение пыталось удалять/выбирать данные из таблиц, которых нет в БД
- Каждая неудачная попытка → запись в syslog → быстрый рост лога

---

### 2.4. Проверка наличия таблиц в PostgreSQL

```bash
sudo -u postgres psql -h 192.168.87.60 -p 5432 -d postgres -c "\dt *.*_dialer_attempt"
```

**Результат:**
```
Did not find any relation named "*.*_dialer_attempt".
```
✅ Подтвердили — таблиц действительно нет

---

### 2.5. Обнаружение системы авто-перезапуска

```bash
sudo pkill -f "runtel_pbx_v2_master/iface.py"   # убили процессы
ps aux | grep runtel_pbx_v2_master               # они снова появились!
```

**Что нашли:**
- Процессы автоматически перезапускаются
- Значит, есть systemd-сервис или supervisor

```bash
systemctl list-units --all | grep -i "runtel\|billing\|pbx"
```

**Результат:**
```
billing.service          → runtel_billing_v2
runtel-cdr-v2.service    → runtel-cdr
runtel-core-v2.service   → runtel-core
runtel-iface-v2.service  → runtel-iface   ← ВИНОВНИК!
```

---

## 3. Итоговое решение

### 3.1. Подавление логов Patroni (быстрое решение)

```bash
sudo tee /etc/rsyslog.d/99-suppress-patroni.conf << 'EOF'
:programname, isequal, "patroni" stop
EOF
sudo systemctl restart rsyslog
```

**Зачем:**  
Даже если Runtel продолжит работать, его ошибки не будут писаться в syslog

### 3.2. Остановка виновного сервиса

```bash
sudo systemctl stop runtel-iface-v2.service
sudo systemctl disable runtel-iface-v2.service   # отключить автозапуск
sudo systemctl mask runtel-iface-v2.service      # защита от случайного запуска
```

**Зачем:**  
Полностью остановить поток ошибок на уровне источника
