# Исправление ошибок и предупреждений Nextcloud 33

## Введение
После установки Nextcloud 33 на Ubuntu 24.04 с Nginx, PostgreSQL, OnlyOffice и Keycloak SSO в интерфейсе администратора (`https://nextcloud.runtel.ru/settings/admin/overview`) могут появляться различные предупреждения и ошибки. Эта инструкция поможет их устранить.

---

## 1. Исправление ошибки getenv() в PHP

### Проблема
PHP не настроен правильно для получения переменных системного окружения. Запрос `getenv("PATH")` возвращает пустые результаты.

### Диагностика
```bash
# Проверяем текущее значение variables_order
sudo -u www-data php -r "echo ini_get('variables_order');"
# Должно быть: EGPCS, а не GPCS

# Проверяем вывод PATH
sudo -u www-data php -r "echo getenv('PATH');"
# Должен выводиться путь, а не пустая строка
```

### Решение

**Шаг 1.** Исправляем основной конфигурационный файл PHP для FPM:
```bash
# Возможно потребуется поискать
grep -r "variables_order" /etc/php/8.3/ | grep -v "EGPCS"

# Редактируем php.ini для FPM
sudo sed -i 's/variables_order = "GPCS"/variables_order = "EGPCS"/' /etc/php/8.3/fpm/php.ini
```

**Пояснение:** Команда `sed` ищет строку `variables_order = "GPCS"` и заменяет её на `variables_order = "EGPCS"`. Буква `E` означает, что переменные окружения (Environment) будут доступны через `$_ENV` и `getenv()`.

**Шаг 2.** Исправляем PHP для командной строки (CLI) на всякий случай:
```bash
sudo sed -i 's/variables_order = "GPCS"/variables_order = "EGPCS"/' /etc/php/8.3/cli/php.ini
```

**Шаг 3.** Перезапускаем PHP-FPM:
```bash
sudo systemctl restart php8.3-fpm
```

**Шаг 4.** Проверяем результат:
```bash
# Проверяем variables_order
sudo -u www-data php -r "echo ini_get('variables_order');"
# Ожидаемый вывод: EGPCS

# Проверяем getenv
sudo -u www-data php -r "echo getenv('PATH');"
# Ожидаемый вывод: /usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/snap/bin
```

## 2. Исправление ошибки OCS Provider

### Проблема
Веб-сервер не настроен должным образом для разрешения пути `"/ocs-provider/"`. Nextcloud не может обработать запросы к этому эндпоинту.

### Решение

**Шаг 1.** Открываем конфигурационный файл Nginx:
```bash
sudo mcedit /etc/nginx/sites-available/nextcloud
```

**Шаг 2.** Добавляем правила для ocs-provider **после** блока `location ^~ /.well-known` и **до** закрывающей скобки `}`:

```nginx
    # Решение для ocs-provider (без PHP)
    location = /ocs-provider {
        return 301 /ocs-provider/;
    }

    location ~ ^/ocs-provider(?:/|$) {
        try_files $uri $uri/ /index.php$request_uri;
    }
```

**Пояснение:**
- `location = /ocs-provider` - точное совпадение URL без слеша
- `return 301 /ocs-provider/` - перенаправление на URL со слешем
- `location ~ ^/ocs-provider(?:/|$)` - регулярное выражение для всех URL, начинающихся с `/ocs-provider`
- `try_files $uri $uri/ /index.php$request_uri` - пытается найти файл, затем папку, затем передаёт запрос в index.php

**Шаг 3.** Проверяем конфигурацию Nginx:
```bash
sudo nginx -t
```
Ожидаемый вывод: `nginx: configuration file /etc/nginx/nginx.conf test is successful`

**Шаг 4.** Перезагружаем Nginx:
```bash
sudo systemctl reload nginx
```

## 3. Исправление заголовков HTTP (X-Robots-Tag и HSTS)

### Проблема
- Заголовок `X-Robots-Tag` не установлен на `noindex, nofollow`
- Заголовок `Strict-Transport-Security` (HSTS) отсутствует

### Решение

**Шаг 1.** Открываем конфигурацию Nginx:
```bash
sudo mcedit /etc/nginx/sites-available/nextcloud
```

**Шаг 2.** Находим блок `# Безопасные заголовки` и исправляем/добавляем строки:

```nginx
    # Безопасные заголовки
    add_header Referrer-Policy "no-referrer" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Download-Options "noopen" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Permitted-Cross-Domain-Policies "none" always;
    add_header X-Robots-Tag "noindex, nofollow" always;     # ← ИЗМЕНИТЬ
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Strict-Transport-Security "max-age=15552000; includeSubDomains; preload" always;  # ← ДОБАВИТЬ
```

**Пояснение:**
- `X-Robots-Tag "noindex, nofollow"` - запрещает поисковым роботам индексировать Nextcloud
- `Strict-Transport-Security` - требует использования HTTPS (max-age=15552000 секунд = 180 дней)
- `always` - добавляет заголовки ко всем ответам, включая ошибки

**Шаг 3.** Проверяем и перезагружаем Nginx:
```bash
sudo nginx -t && sudo systemctl reload nginx
```

## 4. Настройка времени окна обслуживания

### Проблема
Не настроено время начала окна обслуживания. Ресурсоёмкие задачи могут выполняться в часы пик.

### Решение
```bash
sudo -u www-data php /var/www/html/nextcloud/occ config:system:set maintenance_window_start --type=integer --value=1
```

**Пояснение:**
- `maintenance_window_start` - время начала окна обслуживания
- `--type=integer` - тип значения (целое число)
- `--value=1` - значение: 1 = 01:00 ночи (0-23, где 0 = полночь)

## 5. Добавление ID сервера

### Проблема
Идентификатор сервера не настроен. Это важно для работы на нескольких PHP-серверах.

### Решение
```bash
sudo -u www-data php /var/www/html/nextcloud/occ config:system:set server_id --value="nextcloud-$(hostname)-$(date +%s)"
```

**Пояснение:**
- `server_id` - уникальный идентификатор сервера
- `$(hostname)` - подставляет имя хоста
- `$(date +%s)` - подставляет текущую временную метку Unix
- Результат: например, `nextcloud-Nextcloud-1779457124`

## 6. Настройка поддержки SVG в ImageMagick

### Проблема
Модуль PHP "imagick" не имеет поддержки SVG формата.

### Диагностика
```bash
# Проверяем, есть ли поддержка SVG
grep -in "SVG" /etc/ImageMagick-6/policy.xml
```

### Решение

**Шаг 1.** Открываем policy.xml ImageMagick:
```bash
sudo mcedit /etc/ImageMagick-6/policy.xml
```

**Шаг 2.** Добавляем строку с разрешением для SVG (можно добавить после строки 143):
```xml
    <policy domain="coder" rights="read|write" pattern="{SVG,PS,PDF,EPS,AI}" />
```

**Пояснение:**
- `domain="coder"` - применяется к кодекам форматов
- `rights="read|write"` - разрешает чтение и запись
- `pattern="{SVG,PS,PDF,EPS,AI}"` - список разрешённых форматов

**Шаг 3.** Перезапускаем PHP:
```bash
sudo systemctl restart php8.3-fpm
```

## 7. Настройка поддержки OTF шрифтов

### Проблема
Не удалось проверить поддержку загрузки файлов `.otf`. Веб-сервер может не отдавать шрифты правильно.

### Решение

**Шаг 1.** Добавляем MIME тип для OTF в Nginx:
```bash
sudo sed -i '/types {/a \    font/otf otf;' /etc/nginx/mime.types
```

**Пояснение:** Команда `sed` добавляет строку `    font/otf otf;` после строки, содержащей `types {`.

**Шаг 2.** Проверяем, что MIME тип добавлен:
```bash
grep "otf" /etc/nginx/mime.types
```

**Шаг 3.** Перезагружаем Nginx:
```bash
sudo systemctl reload nginx
```

**Шаг 4.** Устанавливаем правильный URL для проверки:
```bash
sudo -u www-data php /var/www/html/nextcloud/occ config:system:set overwrite.cli.url --value="https://nextcloud.runtel.ru"
```

## 8. Отключение некритичных сервисов

### 8.1. Отключение AppAPI (если не используются внешние приложения)
```bash
sudo -u www-data php /var/www/html/nextcloud/occ app:disable app_api
```

### 8.2. Отключение требования двухфакторной аутентификации
```bash
sudo -u www-data php /var/www/html/nextcloud/occ config:system:set twofactor_enforced --value="false"
```

## 9. Очистка логов и финальный ремонт

**Шаг 1.** Очищаем лог ошибок:
```bash
sudo truncate -s 0 /var/www/html/nextcloud/data/nextcloud.log
```
**Пояснение:** `truncate -s 0` обрезает файл до нулевого размера (очищает его).

**Шаг 2.** Запускаем полный ремонт системы:
```bash
sudo -u www-data php /var/www/html/nextcloud/occ maintenance:repair --include-expensive
```

**Пояснение:**
- `maintenance:repair` - запускает процедуры восстановления
- `--include-expensive` - включает ресурсоёмкие операции (миграции mimetype и т.д.)

**Шаг 3.** Запускаем обычный ремонт:
```bash
sudo -u www-data php /var/www/html/nextcloud/occ maintenance:repair
```

**Шаг 4.** Перезапускаем все сервисы:
```bash
sudo systemctl restart php8.3-fpm nginx
```

## 10. Финальная проверка

1. **Очистите кеш браузера** (Ctrl+Shift+Delete) или откройте страницу в режиме инкогнито.

2. **Перейдите на страницу:** `https://nextcloud.runtel.ru/settings/admin/overview`

3. **Ожидаемый результат:** Должны остаться только информационные предупреждения (не критические):
   - AppAPI (если не отключили)
   - Двухфакторная аутентификация
   - Загрузка файла шрифта (может остаться, не критично)

```bash
# 2. Перезапуск PHP
sudo systemctl restart php8.3-fpm

# 3. Настройка окна обслуживания
sudo -u www-data php /var/www/html/nextcloud/occ config:system:set maintenance_window_start --type=integer --value=1

# 4. Настройка ID сервера
sudo -u www-data php /var/www/html/nextcloud/occ config:system:set server_id --value="nextcloud-$(hostname)-$(date +%s)"

# 5. Добавление MIME типа для OTF
sudo sed -i '/types {/a \    font/otf otf;' /etc/nginx/mime.types

sudo -u www-data php /var/www/html/nextcloud/occ app:disable app_api
sudo -u www-data php /var/www/html/nextcloud/occ config:system:set twofactor_enforced --value="false"

# 7. Очистка логов и ремонт

sudo -u www-data php /var/www/html/nextcloud/occ maintenance:repair --include-expensive
sudo -u www-data php /var/www/html/nextcloud/occ maintenance:repair

# 8. Финальный перезапуск
sudo systemctl restart php8.3-fpm nginx
```
