## 👤 Инициализация и создание администратора

### Первоначальная настройка

Перед первым запуском необходимо инициализировать конфигурацию OpenCloud и создать учетную запись администратора.

**Создание конфигурации с автоматическим паролем:**

```bash
cd /root/programs/OpenCloud/Git/opencloud

# Инициализация с автоматической генерацией пароля
./bin/opencloud init --insecure true

# Вывод команды покажет:
# =========================================
#  generated OpenCloud Config
# =========================================
#  configpath : /root/.opencloud/config/opencloud.yaml
#  user       : admin
#  password   : <автоматически сгенерированный пароль>
```

**Создание конфигурации с заданным паролем администратора:**

```bash
# Инициализация с указанием конкретного пароля
./bin/opencloud init --insecure true --admin-password ваш_пароль

# Пример:
./bin/opencloud init --insecure true --admin-password PASSword4

# Результат:
# =========================================
#  generated OpenCloud Config
# =========================================
#  configpath : /root/.opencloud/config/opencloud.yaml
#  user       : admin
#  password   : PASSword4
```

**Принудительная перезапись существующей конфигурации:**

```bash
# Если конфигурация уже существует, используйте флаг --force-overwrite
./bin/opencloud init --insecure true --admin-password новый_пароль --force-overwrite

# Система создаст резервную копию старой конфигурации:
# /root/.opencloud/config/opencloud.yaml.2026-05-05-11-44-43.backup
```

### 📋 Параметры команды init

| Флаг | Описание |
|------|----------|
| `--insecure true` | Разрешить работу по HTTP (только для тестовой среды) |
| `--admin-password` | Указать пароль администратора (иначе генерируется автоматически) |
| `--force-overwrite` | Принудительно перезаписать существующий конфиг с созданием бэкапа |
| `--diff` | Показать различия между текущим и новым конфигом (без перезаписи) |

### 📁 Где хранятся файлы конфигурации

```bash
# Директория конфигурации
/root/.opencloud/config/

# Основной файл конфигурации
/root/.opencloud/config/opencloud.yaml

# Резервные копии (при использовании --force-overwrite)
/root/.opencloud/config/opencloud.yaml.2026-05-05-11-44-43.backup
```

### 🔍 Проверка конфигурации

```bash
# Просмотр созданной конфигурации
cat /root/.opencloud/config/opencloud.yaml

# Просмотр пароля администратора (если был сгенерирован автоматически)
grep password /root/.opencloud/config/opencloud.yaml
```

> **💡 Важно сохранить пароль администратора!** При автоматической генерации пароль выводится только один раз при выполнении команды `init`. При утере пароля потребуется сброс через базу данных или переинициализация с флагом `--force-overwrite`.

## 🔒 Настройка Nginx и HTTPS

### Решение проблемы с SSL-сертификатом

#### Ситуация

При попытке настроить HTTPS для домена `oc.runtel.ru` стандартными методами (Certbot) возникала ошибка:

```
Certbot failed to authenticate some domains (authenticator: nginx). 
The Certificate Authority reported these problems:
Domain: oc.runtel.ru
Type: dns
Detail: no valid A records found for oc.runtel.ru
```

**Проблема:** домен `oc.runtel.ru` на момент настройки не имел прямой A-записи, указывающей на сервер, так как использовался плавающий IP (Teleport).

#### Решение: импорт готового wildcard-сертификата

В распоряжении уже был валидный сертификат для всех поддоменов `*.runtel.ru` в одном файле `runtel.pem`. Этот файл содержал полную цепочку сертификатов и приватный ключ (формат PKCS#12).

**Шаг 1: Разделение файла на сертификат и ключ**

Файл `runtel.pem` содержал:
- `-----BEGIN CERTIFICATE-----` (сертификат)
- `-----BEGIN RSA PRIVATE KEY-----` (приватный ключ)

Для корректной работы Nginx файлы нужно разделить:

```bash
cd /etc/nginx/ssl

# Извлечение сертификата
sed -n '/-----BEGIN CERTIFICATE-----/,/-----END CERTIFICATE-----/p' runtel.pem > oc.runtel.ru.crt

# Извлечение приватного ключа
sed -n '/-----BEGIN RSA PRIVATE KEY-----/,/-----END RSA PRIVATE KEY-----/p' runtel.pem > oc.runtel.ru.key

# Проверка сертификата
cat oc.runtel.ru.crt | openssl x509 -text -noout | grep -E "Subject:|Not After"

# Проверка ключа
cat oc.runtel.ru.key | openssl rsa -check -noout
```

**Шаг 2: Подключение сертификата в Nginx**

```nginx
server {
    listen 443 ssl http2;
    server_name oc.runtel.ru;

    ssl_certificate /etc/nginx/ssl/oc.runtel.ru.crt;
    ssl_certificate_key /etc/nginx/ssl/oc.runtel.ru.key;

    location / {
        proxy_pass http://127.0.0.1:9200;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**Шаг 3: Перезагрузка Nginx**

```bash
systemctl reload nginx
```

#### Почему нельзя было использовать `runtel.pem` напрямую?

Nginx требует раздельных файлов:
- `ssl_certificate` — только сертификат (публичная часть)
- `ssl_certificate_key` — только приватный ключ

Файл `runtel.pem` содержит и то, и другое вместе. Nginx не умеет автоматически разделять такие связки (bundle). Кроме того, приватный ключ должен храниться с правами доступа 600 или 400 — в едином файле этого не обеспечить.

### Дополнительная настройка Nginx для работы с HTTPS-бэкендом

В процессе настройки выяснилось, что OpenCloud (сервер на порту 9200) **требует HTTPS-соединения даже внутри**, а не HTTP. Поэтому пришлось внести изменения в конфигурацию Nginx.

#### 📄 Итоговый рабочий конфиг Nginx

```nginx
server {
    listen 80;
    server_name oc.runtel.ru;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl;
    server_name oc.runtel.ru;
    
    ssl_certificate /etc/nginx/ssl/oc.runtel.ru.crt;
    ssl_certificate_key /etc/nginx/ssl/oc.runtel.ru.key;
    
    location / {
        # Ключевое изменение: проксируем на HTTPS бэкенда
        proxy_pass https://127.0.0.1:9200;
        
        # Отключаем проверку сертификата бэкенда, так как OpenCloud
        # на внутреннем порту 9200 использует свой собственный (самоподписанный) сертификат.
        # Внешний сертификат (для браузера) при этом остаётся валидным.
        proxy_ssl_verify off;

        # Передаём правильные заголовки
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Forwarded-Host $host;
        proxy_set_header X-Forwarded-Port 443;
    }
}
```

**Зачем это нужно:**
- `proxy_pass https://127.0.0.1:9200` — заставляет Nginx общаться с OpenCloud по HTTPS.
- `proxy_ssl_verify off` — отключает проверку сертификата бэкенда (иначе Nginx будет ругаться на самоподписанный сертификат OpenCloud).
- Дополнительные заголовки `X-Forwarded-*` помогают OpenCloud правильно определять реальный протокол и адрес клиента.

<details>
<summary>📖 ПОДРОБНОЕ ПОЯСНЕНИЕ ПРО СЕРТИФИКАТЫ (нажмите, чтобы развернуть)</summary>

**У нас валидный сертификат от Let's Encrypt (wildcard `*.runtel.ru`)** — это отличный, доверенный сертификат.

Тогда зачем нужен `proxy_ssl_verify off`?

`proxy_ssl_verify off` относится **не к сертификату, который видят браузеры**, а к **сертификату самого OpenCloud на порту 9200**.

Дело в том, что **OpenCloud внутри себя (на порту 9200) использует СВОЙ собственный сертификат** — он генерируется автоматически при инициализации и является **самоподписанным**.

У нас есть два слоя:

1. **Внешний слой (Nginx → браузер)** — использует ваш валидный сертификат `oc.runtel.ru.crt`. Здесь всё отлично, браузер показывает зелёный замок.

2. **Внутренний слой (Nginx → OpenCloud на порту 9200)** — OpenCloud сам себе генерирует сертификат (обычно самоподписанный). Nginx, когда проксирует запросы на `https://127.0.0.1:9200`, проверяет этот внутренний сертификат. И поскольку он самоподписанный, Nginx по умолчанию его отвергает.

Именно для этого и нужна директива `proxy_ssl_verify off`. Она говорит Nginx: **«Не проверяй сертификат бэкенда (OpenCloud), доверяй ему без проверки»**.

Если бы мы попытались подключиться без этой директивы, Nginx выдал бы ошибку:

```
502 Bad Gateway
SSL certificate problem: self signed certificate
```

**Как исправить «по-красивому» (опционально):**

Если вы хотите, чтобы и внутреннее соединение было полностью валидным, можно заменить самоподписанный сертификат OpenCloud на ваш валидный. Но это требует правок в конфиге OpenCloud:

```yaml
# В /root/.opencloud/config/opencloud.yaml
opencloud:
  http:
    tls_cert: /etc/nginx/ssl/oc.runtel.ru.crt
    tls_key: /etc/nginx/ssl/oc.runtel.ru.key
```

Тогда `proxy_ssl_verify off` станет не нужен, и можно будет включить проверку:

```nginx
proxy_ssl_verify on;
proxy_ssl_trusted_certificate /etc/nginx/ssl/oc.runtel.ru.crt;
```

Но для тестовой среды и для простоты **директивы `proxy_ssl_verify off` достаточно** — внутренний трафик всё равно идёт через localhost.

</details>

### Обход DNS через файл hosts (для локального тестирования)

Из-за особенностей сетевой инфраструктуры (наличие сервера Teleport, который перехватывал запросы к домену `oc.runtel.ru` и перенаправлял их на себя) пришлось временно добавить запись в **локальный файл hosts на компьютере пользователя**:

```hosts
192.168.87.7 oc.runtel.ru
```

**Где находится файл hosts:**
- **Windows**: `C:\Windows\System32\drivers\etc\hosts`
- **Linux / Mac**: `/etc/hosts`

**Зачем это нужно:**
- Браузер теперь обращается напрямую к серверу `192.168.87.7`, игнорируя публичную DNS (которая указывала на IP Teleport).
- Это позволяет протестировать работу OpenCloud до тех пор, пока не будет настроена правильная маршрутизация на балансировщике или в Teleport.

> **⚠️ Важно:** Эта запись нужна **только для тестирования** и только на компьютере, с которого вы открываете браузер. На самом сервере OpenCloud её добавлять не нужно.

Может потребоваться, если не можете войти https://oc.runtel.ru/signin/v1/welcome


## 📌 Итоговые замечания

1. **Основная проблема** — проект находится в стадии перехода от OwnCloud к OpenCloud, многие зависимости остались с префиксом `owncloud`

2. **Ключевые инструменты**, которые пришлось устанавливать вручную:
   - `buf` (для protobuf)
   - `pigeon` (для парсера грамматики)
   - `bingo` (для управления зависимостями)

3. **Временное отключение сервисов** (`search` и `settings`) не влияет на базовую функциональность файлового хранилища

4. **Успешная сборка** заняла около 2 часов из-за необходимости решения всех проблем

5. **Бинарник** получился размером 150 МБ, что нормально для Go приложения

6. **При инициализации** обязательно используйте флаг `--insecure true` для тестовой среды, иначе OpenCloud будет требовать HTTPS

7. **Wildcard-сертификат** `*.runtel.ru` успешно импортирован и работает для домена `oc.runtel.ru`

8. **Nginx выступает в роли HTTPS-прокси** перед OpenCloud: принимает HTTPS-запросы от браузера и передаёт их на внутренний HTTPS-порт OpenCloud, отключая проверку сертификата бэкенда

9. **Для локального тестирования** пришлось добавить запись в `hosts`, чтобы обойти перехват трафика сервером Teleport. В боевой среде необходимо настроить балансировщик или Teleport для правильной маршрутизации домена `oc.runtel.ru`

10. **Финальный доступ:** веб-интерфейс доступен по адресу `https://oc.runtel.ru/apps/web`, страница входа — `https://oc.runtel.ru/signin/v1/identifier`
