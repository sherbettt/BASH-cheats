## Подготовительные работы

### Важное правило: пользователи в PostgreSQL
В PostgreSQL используется аутентификация `peer` - системный пользователь должен соответствовать пользователю БД. Поэтому все операции нужно выполнять от пользователя `postgres`.

```bash
# Переключиться на пользователя postgres (вариант 1)
su - postgres

# Переключиться на пользователя postgres (вариант 2 - от root)
sudo -u postgres bash

# Переключиться на пользователя postgres (вариант 3 - выполнить одну команду)
sudo -u postgres <команда>
```

### Просмотр существующих баз данных
```bash
# Показать все базы данных (вариант 1 - в psql)
psql -l

# Показать все базы данных (вариант 2 - в psql интерактивно)
psql
\l

# Показать конкретную базу
psql -l | grep jira

# В psql показать информацию о конкретной базе
\l jira_restored
```

## Создание дампа на исходном сервере

### Типы дампов PostgreSQL

| Тип дампа | Команда | Описание |
|-----------|---------|----------|
| Полный (структура + данные) | `pg_dump jira_restored > jira.sql` | Самый распространённый |
| Только структура | `pg_dump --schema-only jira_restored > schema.sql` | Без данных |
| Только данные | `pg_dump --data-only jira_restored > data.sql` | Только INSERT-ы |
| Сжатый формат | `pg_dump -Fc jira_restored > jira.dump` | Для pg_restore |
| С INSERT-ами | `pg_dump --inserts jira_restored > jira.sql` | Более совместимо |
| С именами колонок | `pg_dump --column-inserts jira_restored > jira.sql` | Максимальная совместимость |

### Создание дампа Jira
```bash
# Переключиться на postgres
su - postgres

# Создать дамп в сжатом формате (рекомендуется)
pg_dump -Fc jira_restored > /tmp/jira_backup_$(date +%Y%m%d).dump

# Пояснение ключей:
# -Fc - сжатый формат (custom), совместим с pg_restore
# $(date +%Y%m%d) - подставит текущую дату в формате ГГГГММДД
# > /tmp/... - перенаправление вывода в файл

# Альтернативы:

# 1. Обычный SQL дамп (читаемый, но больше размер)
pg_dump jira_restored > /tmp/jira_backup_$(date +%Y%m%d).sql

# 2. Сжатый gz (экономит место)
pg_dump jira_restored | gzip > /tmp/jira_backup_$(date +%Y%m%d).sql.gz

# 3. С подробным выводом процесса
pg_dump --verbose -Fc jira_restored > /tmp/jira_backup_$(date +%Y%m%d).dump

# 4. Без владельцев (полезно при переносе)
pg_dump --no-owner -Fc jira_restored > /tmp/jira_backup_$(date +%Y%m%d).dump

# Проверка созданного дампа
ls -lh /tmp/jira_backup_*.dump
# -rw-r--r-- 1 postgres postgres 782M мар 10 10:27 jira_backup_20260310.dump

# Просмотр содержимого дампа (первые строки)
head -20 /tmp/jira_backup_20260310.dump

# Для SQL дампа можно посмотреть наличие данных
grep -i "project" /tmp/jira_backup_*.sql | head -5
```

## Копирование дампа на целевой сервер

### Вариант 1: SCP (Secure Copy)
```bash
# От root на исходном сервере
scp /tmp/jira_backup_20260310.dump root@192.168.45.201:/tmp/

# С указанием порта SSH (если нестандартный)
scp -P 22 /tmp/jira_backup_20260310.dump root@192.168.45.201:/tmp/

# С сохранением прав и временных меток
scp -p /tmp/jira_backup_20260310.dump root@192.168.45.201:/tmp/
```

### Вариант 2: Rsync (рекомендуется для больших файлов)
```bash
# Rsync показывает прогресс и может докачивать
rsync -avP /tmp/jira_backup_20260310.dump root@192.168.45.201:/tmp/

# Пояснение ключей:
# -a - archive mode (сохраняет права, ссылки и т.д.)
# -v - verbose (подробный вывод)
# -P - показывать прогресс и поддержка докачки
```

### Вариант 3: По частям (при нестабильной сети)
```bash
# На исходном сервере - разделить на части по 100MB
split -b 100M /tmp/jira_backup_20260310.dump /tmp/jira_backup_part_

# Скопировать все части
scp /tmp/jira_backup_part_* root@192.168.45.201:/tmp/

# На целевом сервере - собрать обратно
cat /tmp/jira_backup_part_* > /tmp/jira_backup_20260310.dump
rm /tmp/jira_backup_part_*  # удалить части после сборки
```

### Проверка копирования
```bash
# На целевом сервере проверить размер
ls -lh /tmp/jira_backup_20260310.dump

# Проверить контрольную сумму (на обоих серверах)
md5sum /tmp/jira_backup_20260310.dump
# Сравнить хеш-суммы - должны совпадать
```

```bash
# | Member | Host           | Role    | State   | TL | Lag in MB |
# +--------+----------------+---------+---------+----+-----------+
# | pg1    | 192.168.45.201 | Leader  | running | 45 |           |
# | pg2    | 192.168.45.202 | Replica | running | 45 |         0 |
# | pg3    | 192.168.45.204 | Replica | running | 45 |         0 |
# +--------+----------------+---------+---------+----+-----------+

# Если не указан конфиг, patronictl может ругаться
patronictl list  # может выдать WARNING

# Проверить конкретный узел
patronictl -c /etc/patroni.yml show-config

# Проверить, кто лидер (мастер)
patronictl -c /etc/patroni.yml list | grep Leader

# Альтернативный способ - через psql
psql -c "SELECT pg_is_in_recovery();"
# f = false = это мастер (leader)
# t = true = это реплика
```

### Важно для Patroni
- **Все операции записи (восстановление) делаются ТОЛЬКО на лидере (мастере)**
- Реплики синхронизируются автоматически
- Если сделать восстановление на реплике - данные не синхронизируются

```bash
# Вариант 1 - от пользователя postgres
su - postgres
psql

# Вариант 2 - через sudo
sudo -u postgres psql

# Вариант 3 - сразу к конкретной базе
```

### Работа с базами данных в psql
```sql
-- Просмотр всех баз
\l

-- Просмотр конкретной базы
\l jira_restored

-- Создание новой базы
CREATE DATABASE jira_reserve OWNER jira;

-- Пояснение: OWNER jira - владельцем базы будет пользователь jira

-- Альтернативы создания:
CREATE DATABASE jira_reserve;  -- владельцем будет текущий пользователь
CREATE DATABASE jira_reserve WITH OWNER = jira ENCODING = 'UTF8';

-- Удаление базы
DROP DATABASE IF EXISTS jira_reserve;

-- Принудительное удаление (если есть подключения)
SELECT pg_terminate_backend(pid) 
FROM pg_stat_activity 
WHERE datname = 'jira_reserve' AND pid <> pg_backend_pid();

DROP DATABASE jira_reserve;

-- Переименование базы
ALTER DATABASE jira_reserve RENAME TO jira_reserve_old;

-- Просмотр размера базы
SELECT pg_size_pretty(pg_database_size('jira_reserve'));
```

## Восстановление из дампа

### Анализ дампа перед восстановлением
```bash
# Посмотреть структуру дампа (что внутри)
pg_restore --list /tmp/jira_backup_20260310.dump | head -50

# Сохранить список в файл
pg_restore --list /tmp/jira_backup_20260310.dump > /tmp/dump_contents.txt

# Посмотреть, от какой версии PostgreSQL дамп
pg_restore --list /tmp/jira_backup_20260310.dump | grep "Dumped from database version"
```

### Восстановление из сжатого дампа (-Fc)
```bash
# Базовая команда восстановления
pg_restore -d jira_reserve /tmp/jira_backup_20260310.dump

# Пояснение: -d jira_reserve - база назначения

# С подробным выводом (рекомендуется)
pg_restore --verbose -d jira_reserve /tmp/jira_backup_20260310.dump

# С очисткой перед созданием
pg_restore --clean --if-exists -d jira_reserve /tmp/jira_backup_20260310.dump

# Пояснение:
# --clean - удалять объекты перед созданием
# --if-exists - использовать IF EXISTS для удаления

# С игнорированием ошибок (не рекомендуется, но иногда нужно)
pg_restore --exit-on-error -d jira_reserve /tmp/jira_backup_20260310.dump || true

# Восстановить только данные
pg_restore --data-only -d jira_reserve /tmp/jira_backup_20260310.dump

# Восстановить только схему
pg_restore --schema-only -d jira_reserve /tmp/jira_backup_20260310.dump

# Восстановить с увеличением памяти
PGOPTIONS='-c maintenance_work_mem=2GB' pg_restore -d jira_reserve /tmp/jira_backup_20260310.dump
```

### Восстановление из SQL дампа
```bash
# Если дамп в формате plain SQL (не сжатый)
psql -d jira_reserve -f /tmp/jira_backup_20260310.sql

# С подробным выводом
psql -d jira_reserve -f /tmp/jira_backup_20260310.sql 2>&1 | tee /tmp/restore.log

# Если дамп сжатый gz
gunzip -c /tmp/jira_backup_20260310.sql.gz | psql -d jira_reserve

# С игнорированием ошибок
psql -d jira_reserve -f /tmp/jira_backup_20260310.sql 2>/dev/null || true
```

### Конвертация сжатого дампа в SQL
```bash
# Конвертировать сжатый дамп в SQL файл
pg_restore --file=/tmp/jira_restore.sql /tmp/jira_backup_20260310.dump

# Затем восстановить через psql
psql -d jira_reserve -f /tmp/jira_restore.sql

# Это может быть полезно, если pg_restore падает с ошибками
```

## Диагностика и устранение проблем

### Проблема 1: "role 'root' does not exist"
**Ошибка:** Выполнение команд от root
```bash
pg_restore: ошибка: подключиться к серверу через сокет не удалось: FATAL:  role "root" does not exist
```

**Решение:** Всегда выполняйте команды от postgres
```bash
# Правильно:
sudo -u postgres pg_restore -d jira_reserve /tmp/jira_backup_20260310.dump

# Или:
su - postgres
pg_restore -d jira_reserve /tmp/jira_backup_20260310.dump
```

### Проблема 2: "No space left on device"
**Ошибка:** Закончилось место на диске
```bash
could not create relation-cache initialization file: No space left on device
```

**Диагностика:**
```bash
# Проверить свободное место
df -kh

# Найти большие директории
du -sh /* 2>/dev/null | sort -rh | head -20

# Проверить данные PostgreSQL
du -sh /var/lib/postgresql/*

# Проверить WAL файлы
du -sh /var/lib/postgresql/patroni/pg_wal
```

**Решение:**
```bash
# Очистить apt кэш
apt clean
apt autoremove --purge -y

# Очистить старые логи
journalctl --vacuum-size=500M

# Удалить ненужные дампы
rm -f /tmp/*.sql /tmp/*.dump  # осторожно, только ненужные!

# Если проблема в WAL файлах PostgreSQL:
CHECKPOINT;  -- в psql

# Или через Patroni
patronictl -c /etc/patroni.yml restart pgcluster pg1 --force
```

### Проблема 3: "server closed the connection unexpectedly"
**Ошибка:** PostgreSQL крашится при восстановлении

**Диагностика:**
```bash
# Посмотреть логи PostgreSQL
tail -f /var/log/postgresql/postgresql-*.log

# Проверить ресурсы
free -h
df -kh
```

**Решение:**
```bash
# Увеличить параметры памяти для восстановления
# Отредактировать postgresql.conf
vim /etc/postgresql/15/main/postgresql.conf

# Добавить:
shared_buffers = 2GB
work_mem = 256MB
maintenance_work_mem = 2GB
max_wal_size = 4GB
checkpoint_timeout = 30min

# Перезапустить PostgreSQL
systemctl restart postgresql
# Или для Patroni:
patronictl restart pgcluster pg1
```

### Проблема 4: Таблицы уже существуют
```bash
pg_restore: ошибка: could not execute query: ERROR:  relation "AO_0201F0_KB_HELPFUL_AGGR" already exists
```

**Решение:**
```bash
# Очистить базу и восстановить с флагом --clean
DROP DATABASE IF EXISTS jira_reserve;
CREATE DATABASE jira_reserve OWNER jira;
pg_restore --clean --if-exists -d jira_reserve /tmp/jira_backup_20260310.dump
```

### Проблема 5: Patroni не отвечает
```bash
# Проверить статус Patroni
systemctl status patroni

# Проверить etcd (хранилище конфигурации Patroni)
curl http://192.168.45.201:2379/version

# Перезапустить Patroni
systemctl restart patroni
```

# 🔌 ПРОЦЕСС ПОДКЛЮЧЕНИЯ БАЗЫ ДАННЫХ К JIRA

## **Что мы сделали шаг за шагом:**

### **Шаг 1: Подготовка - остановка Jira**

Перед изменением конфигурации Jira нужно остановить, чтобы избежать конфликтов и повреждения данных.

```bash
# Остановили Jira через скрипт (ручной запуск)
/opt/atlassian/jira/bin/stop-jira.sh

# Убедились, что процесс остановился
ps aux | grep jira
```

**Почему:** Jira держит постоянное соединение с базой данных. Если менять конфигурацию на работающей Jira, могут возникнуть ошибки.

---
