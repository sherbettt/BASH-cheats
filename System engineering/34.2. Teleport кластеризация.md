# Кластеризация Teleport

## Введение

В этой статье описан практический опыт развёртывания отказоустойчивого кластера Teleport Enterprise (версия 15.4.29) на инфраструктуре Runtel Corp. Рассмотрены два варианта кластеризации: горизонтальное масштабирование Proxy-сервисов и полная отказоустойчивость с использованием PostgreSQL.

### Исходная инфраструктура

| Компонент | IP-адрес | Роль |
|-----------|----------|------|
| **teleport1** | 192.168.87.238 | Auth + Proxy |
| **teleport2** | 192.168.87.239 | Будет добавлен как Proxy |
| **pg1** | 192.168.45.201 | PostgreSQL (Patroni) |
| **pg2** | 192.168.45.202 | PostgreSQL (Patroni) |
| **pg3** | 192.168.45.204 | PostgreSQL (Patroni) |
| **HAProxy** | 192.168.45.201 (pg1) | Балансировщик |

---

## Варианты кластеризации

| Вариант | Сложность | Для каких задач |
|---------|-----------|-----------------|
| **Только Proxy HA** | Низкая | Балансировка веб-трафика (можно добавить второй Proxy) |
| **Auth HA** | Средняя | Отказоустойчивость центра аутентификации (требует внешнюю БД) |
| **Полный кластер** | Высокая | Enterprise-сценарии, большие нагрузки |

### Рекомендации

- **Если нужна отказоустойчивость Proxy** — добавь второй Proxy сервер и поставь перед ними балансировщик (HAProxy/Nginx). Это просто и не требует переделки Auth.
- **Если нужна полная отказоустойчивость** — разворачивай PostgreSQL для Auth, добавляй второй Auth сервер, настраивай etcd.
- **Для текущей инфраструктуры** — изначально был один Proxy (`192.168.87.238`). При росте нагрузки или для отказоустойчивости добавляем второй.

---

## Часть 1. Вариант с отказоустойчивостью Proxy

В этом варианте мы добавляем второй Proxy-сервер, настраиваем HAProxy для балансировки трафика между двумя Proxy. Auth Service остаётся на первом сервере.

### 1.1. Создание второго Proxy (teleport2)

Создаём копию первого контейнера teleport1:

```bash
# На PVE-хосте
pct clone <ID_teleport1> <новый_ID> --hostname teleport2 --full
```

Настраиваем IP-адрес на teleport2:

```bash
# В конфиге LXC изменяем IP
mcedit /etc/network/interfaces
# Устанавливаем 192.168.87.239
```

### 1.2. Конфигурация teleport2 как Proxy-only

Файл `/etc/teleport.yaml`:

```yaml
version: v3
teleport:
  auth_token: "9003d85ca27b21a25c50eeb3db16373c"   # токен от teleport1
  ca_pin: ######
  nodename: teleport2.runtel.org
  advertise_ip: 192.168.87.239
  cache:
    type: in-memory
  log:
    output: stderr
    severity: DEBUG
  data_dir: /var/lib/teleport
  storage:
    type: dir
    path: /var/lib/teleport/backend
  auth_server: 192.168.87.238:3025   # ссылка на основной Auth

auth_service:
  enabled: no                         # отключаем Auth

proxy_service:
  enabled: yes
  listen_addr: 0.0.0.0:3023
  tunnel_listen_addr: 0.0.0.0:3080
  web_listen_addr: 0.0.0.0:443
  public_addr: teleport.runtel.org:443
  ssh_public_addr: teleport.runtel.org:3023
  tunnel_public_addr: teleport.runtel.org:443
  https_keypairs:
    - cert_file: /etc/letsencrypt/live/teleport.runtel.org/fullchain.pem
      key_file: /etc/letsencrypt/live/teleport.runtel.org/privkey.pem

ssh_service:
  enabled: no

windows_desktop_service:
  enabled: no

app_service:
  enabled: no
```

### 1.3. Создание токена на teleport1 для присоединения Proxy

```bash
tctl tokens add --type=proxy
```

Вывод:

```
The invite token: 9003d85ca27b21a25c50eeb3db16373c
```

Этот токен нужно вставить в `auth_token` в конфиге teleport2.

### 1.4. Запуск teleport2

```bash
systemctl restart teleport
journalctl -u teleport -f
```

Проверяем, что порты слушаются:

```bash
ss -tlnp | grep -E "443|3023|3080"
```

### 1.5. Настройка HAProxy (балансировщика)

Устанавливаем HAProxy на pg1 (или отдельный сервер):

```bash
apt install haproxy -y
```

Конфигурация `/etc/haproxy/haproxy.cfg`:

```haproxy
global
    log /dev/log local0
    maxconn 4096

defaults
    log global
    mode tcp
    option tcplog
    timeout connect 5s
    timeout client 30s
    timeout server 30s

frontend teleport-web
    bind *:443
    mode tcp
    default_backend teleport-proxy-web

frontend teleport-ssh
    bind *:3023
    mode tcp
    default_backend teleport-proxy-ssh

frontend teleport-tunnel
    bind *:3080
    mode tcp
    default_backend teleport-proxy-tunnel

backend teleport-proxy-web
    mode tcp
    balance roundrobin
    option tcp-check
    server teleport1 192.168.87.238:443 check
    server teleport2 192.168.87.239:443 check

backend teleport-proxy-ssh
    mode tcp
    balance roundrobin
    option tcp-check
    server teleport1 192.168.87.238:3023 check
    server teleport2 192.168.87.239:3023 check

backend teleport-proxy-tunnel
    mode tcp
    balance roundrobin
    option tcp-check
    server teleport1 192.168.87.238:3080 check
    server teleport2 192.168.87.239:3080 check
```

Запускаем HAProxy:

```bash
systemctl restart haproxy
systemctl status haproxy
```

### 1.6. Проверка балансировки

Смотрим статистику HAProxy:

```
http://192.168.45.201:7000
```

Проверяем доступность Teleport через HAProxy:

```bash
curl -k https://192.168.45.201:443
```

## Часть 2. Полная отказоустойчивость с PostgreSQL

В этом варианте Auth Service также кластеризуется с использованием общей PostgreSQL базы данных.

### 2.1. Подготовка PostgreSQL (Patroni)

У нас уже есть кластер Patroni из трёх нод:

| Нода | IP | Роль |
|------|----|----|
| pg1 | 192.168.45.201 | Leader |
| pg2 | 192.168.45.202 | Replica |
| pg3 | 192.168.45.204 | Replica |

#### Создание базы и пользователя для Teleport

Заходим в psql
```
psql -h 192.168.45.201 -p 5000 -U postgres -c "\dt"
psql -h 192.168.45.201 -p 5000 -U postgrespsql -h 192.168.45.201 -p 5000 -U postgres -c "\dt"
```
```sql
CREATE USER teleport_user WITH PASSWORD 'RuntelTeleport2026!';
CREATE DATABASE teleport OWNER teleport_user;
GRANT ALL PRIVILEGES ON DATABASE teleport TO teleport_user;
```

### 2.2. Настройка HAProxy для PostgreSQL

В `/etc/haproxy/haproxy.cfg` уже были настроены фронтенды для PostgreSQL:

```haproxy
frontend postgres_frontend
    bind *:5000
    mode tcp
    default_backend postgres_master

backend postgres_master
    mode tcp
    balance roundrobin
    option httpchk GET /master
    http-check expect status 200
    server pg1 192.168.45.201:5432 check port 8008 inter 5s rise 2 fall 3
    server pg2 192.168.45.202:5432 check port 8008 inter 5s rise 2 fall 3
    server pg3 192.168.45.204:5432 check port 8008 inter 5s rise 2 fall 3
```

Таким образом, Teleport подключается к порту 5000, а HAProxy направляет запросы на актуального мастера PostgreSQL.

### 2.3. Переключение teleport1 на PostgreSQL

Останавливаем teleport1:

```bash
systemctl stop teleport
rm -rf /var/lib/teleport/backend/*
```

В `/etc/teleport.yaml` меняем секцию `auth_service.storage`:

```yaml
auth_service:
  enabled: yes
  listen_addr: 0.0.0.0:3025
  cluster_name: teleport-runtel
  storage:
    type: postgres
    conn_string: "postgres://teleport_user:RuntelTeleport2026!@192.168.45.201:5000/teleport?sslmode=disable"
  authentication:
    type: local
    second_factor: off
```

Запускаем teleport1:

```bash
systemctl start teleport
journalctl -u teleport -f
```

### 2.4. Настройка teleport2 (второй Auth, опционально)

При необходимости teleport2 тоже можно сделать Auth-сервером, указав ту же `conn_string` и включив `auth_service.enabled: yes`.

```bash
mcedit /etc/teleport.yaml
# Включаем auth_service и указываем conn_string
systemctl restart teleport
```

### 2.5. Проверка

```bash
tctl status
tctl users ls
```

Проверяем, что таблицы создались в PostgreSQL:

```sql
\c teleport
\dt
```

Должны быть таблицы: `users`, `roles`, `nodes` и др.

## Заключение

В результате мы получили:

- **Отказоустойчивый Proxy слой** (два Proxy-сервера и HAProxy)
- **Централизованное хранилище Auth** (PostgreSQL кластер)
- **Возможность дальнейшего масштабирования Auth** (добавление второго Auth-сервера)

### Итоговая архитектура

```
                    ┌─────────────────────────────────────┐
                    │         HAProxy (на pg1)             │
                    │  - порт 5000: PostgreSQL master     │
                    │  - порт 443/3023/3080: Teleport    │
                    └─────────────────────────────────────┘
                                   │
                    ┌──────────────┼──────────────┐
                    │              │              │
                    ▼              ▼              ▼
              ┌──────────┐   ┌──────────┐   ┌──────────┐
              │teleport1 │   │teleport2 │   │ Postgres │
              │Auth+Proxy│   │Proxy     │   │ Cluster  │
              │.238      │   │.239      │   │.201/.202│
              └──────────┘   └──────────┘   └─────────┴──┘
```

**Примечание:** При использовании PostgreSQL необходимо следить за:

- Мониторингом Patroni (через API на порту 8008)
- Ротацией паролей пользователя `teleport_user`
