# Полное руководство по развертыванию HA-кластера Teleport Enterprise v16

В данном руководстве описан процесс настройки отказоустойчивого кластера Teleport с использованием внешней БД PostgreSQL, балансировщика HAProxy и интеграции с Keycloak (OIDC).

---

## 1. Подготовка инфраструктуры

| Узел | IP-адрес | Роль |
|-----------|----------|------|
| **teleport1** | 192.168.87.238 | Auth + Proxy + App |
| **teleport2** | 192.168.87.239 | Auth + Proxy + App |
| **pg1 (Leader)** | 192.168.45.201 | PostgreSQL (порт 5000 через HAProxy) |
| **pg2 (Replica)** | 192.168.45.202 | PostgreSQL (реплика) |
| **pg3 (Replica)** | 192.168.45.204 | PostgreSQL (реплика) |
| **HAProxy** | 192.168.45.201 | Балансировщик (порт 5000 для БД, порты 443/3023/3080 для Teleport) |

## 2. Настройка базы данных PostgreSQL

Teleport v16 требует логической репликации для синхронизации кэша между нодами.

### 2.1. Установка wal2json (обязательно!)

```bash
# На всех PostgreSQL серверах
apt install postgresql-15-wal2json
systemctl restart postgresql
```

### 2.2. Создание БД и пользователя

Зайдите на мастер-ноду PostgreSQL и выполните:

```bash
sudo -u postgres psql
```

```sql
-- Создаем пользователя
CREATE USER teleport_user WITH PASSWORD 'VeryStrong77!';

-- Создаем базу данных
CREATE DATABASE teleport OWNER teleport_user;

-- Даем права
GRANT ALL PRIVILEGES ON DATABASE teleport TO teleport_user;

-- ВАЖНО: Для работы Replication Slots (Change Feed) в v16 Enterprise
-- пользователю требуются права SUPERUSER или REPLICATION
ALTER USER teleport_user WITH REPLICATION;

-- Проверяем созданных пользователей
\du
```

**Вывод команды `\du`:**

```
                                             Список ролей
   Имя роли    |                                Атрибуты                                 | Член ролей 
---------------+-------------------------------------------------------------------------+------------
 jira          |                                                                         | {}
 keycloak      |                                                                         | {}
 postgres      | Суперпользователь, Создаёт роли, Создаёт БД, Репликация, Пропускать RLS | {}
 replicator    | Репликация                                                              | {}
 rt_pbx        |                                                                         | {}
 teleport_user | Репликация                                                              | {}
```

### 2.3. Настройка .pgpass для автоматизации

Чтобы утилиты могли подключаться к БД без ввода пароля:

```bash
# На всех серверах (teleport1, teleport2, pg1)
cat >> ~/.pgpass <<EOF
192.168.45.201:5000:teleport:teleport_user:VeryStrong77!
EOF
chmod 600 ~/.pgpass
```

### 2.4. Установка прав на файл .pgpass

Права должны быть строго `600`, иначе PostgreSQL игнорирует файл:

```bash
chmod 600 ~/.pgpass
```

```bash
psql -h 192.168.45.201 -p 5000 -U teleport_user -d teleport -c "\dt"
```

**Ожидаемый вывод:**


## 4. Настройка OIDC (Keycloak)

В v16 коннекторы нельзя описывать в `teleport.yaml`. Их нужно создавать как динамические ресурсы.

### 4.1. Создание манифеста коннектора

```bash
mkdir -p /etc/teleport.d/
mcedit /etc/teleport.d/keycloak-connector.yaml
```

**Содержимое `keycloak-connector.yaml`:**

```yaml
kind: oidc
version: v3
metadata:
  name: keycloak
spec:
  client_id: teleport
  client_secret: "HgO2LUuSb7zKvtxgL2sG5aEErjp1ddop"
  issuer_url: "https://sso.runtel.ru:8443/realms/runtel"
  redirect_url: "https://teleport.runtel.org/v1/webapi/oidc/callback"
  scope: ["openid", "profile", "email", "groups"]
  claims_to_roles:
    - claim: groups
      value: jira-administrators
      roles: ["access", "editor"]
    - claim: groups
      value: confluence-administrators
      roles: ["access", "editor"]
    - claim: groups
      value: jenkins_admin
      roles: ["access", "editor"]
    - claim: email
      value: "k@runtel.ru"
      roles: ["access", "editor"]
    - claim: groups
      value: engineers
      roles: ["access"]

    # Инженеры и пользователи (только доступ)
    - claim: groups
      value: engineers
      roles: ["access"]
    - claim: groups
      value: support
      roles: ["access"]
    - claim: groups
      value: jira-software-users
      roles: ["access"]
    - claim: groups
      value: confluence-users
      roles: ["access"]
    - claim: groups
      value: jira-servicedesk-users
      roles: ["access"]
    - claim: groups
      value: default-roles-runtel
      roles: ["access"]
```

### 4.2. Применение коннектора

```bash
tctl create -f /etc/teleport.d/keycloak-connector.yaml
```

### 4.3. Проверка создания пользователя после входа

```bash
tctl get users
```

**Ожидаемый вывод (появление пользователя из Keycloak):**

```yaml
  name: k@runtel.ru

  created_by:

  - access
```

## 6. Балансировка через HAProxy

### 6.1. Конфигурация HAProxy для PostgreSQL и Teleport

```bash
mcedit /etc/haproxy/haproxy.cfg
```
Добавить к существующей конфигурации с pg кластером
```haproxy
## TELEPORT
# from /etc/teleport.yaml

frontend teleport-web
    bind *:443
    mode tcp
    default_backend teleport-proxy-web

frontend teleport-ssh
    bind *:3023
    mode tcp
    default_backend teleport-proxy-ssh

frontend teleport-tunnel
    bind *:3080
    mode tcp
    default_backend teleport-proxy-tunnel

frontend teleport-auth
    bind *:3025
    mode tcp
    default_backend teleport-auth-backend

backend teleport-proxy-web
    mode tcp
    balance source
    option tcp-check
    #server teleport1 192.168.87.238:443 check #Teleport1 original IP
    #server teleport2 192.168.87.239:443 check #Teleport2 original IP 
    server teleport-vip 192.168.87.240:443 check #Virtual IP keepalived

backend teleport-proxy-ssh
    mode tcp
    balance source
    option tcp-check
    #server teleport1 192.168.87.238:3023 check
    #server teleport2 192.168.87.239:3023 check
    server teleport-vip 192.168.87.240:443 check #Virtual IP keepalived

backend teleport-proxy-tunnel
    mode tcp
    balance source
    option tcp-check
    #server teleport1 192.168.87.238:3080 check
    #server teleport2 192.168.87.239:3080 check
    server teleport-vip 192.168.87.240:443 check

backend teleport-auth-backend
    mode tcp
    balance source
    #server teleport1 192.168.87.238:3025 check
    #server teleport2 192.168.87.239:3025 check
    server teleport-vip 192.168.87.240:443 check 
```

### 6.2. Запуск HAProxy

```bash
haproxy -c -f /etc/haproxy/haproxy.cfg
systemctl restart haproxy
```
Статистика по адресу http://192.168.45.201:7000/

## 9. Типичные проблемы и решения

| Проблема | Решение |
|----------|---------|
| `field oidc not found in type config.Auth` | В v16 OIDC коннектор создаётся через отдельный файл, не в `teleport.yaml` |
| `must be superuser or replication role` | Выдайте пользователю права `ALTER USER teleport_user WITH REPLICATION;` |
| `could not access file "wal2json"` | Установите пакет `postgresql-15-wal2json` |
| `second_factor cannot be off` | Установите `second_factor: on` и настройте `webauthn` |
| `listen tcp 192.168.87.238:3025: bind: cannot assign requested address` | Исправьте `listen_addr` и `public_addr` на IP текущего сервера |
| `redirect_uri mismatch` | Добавьте в Keycloak точный URL: `https://teleport.runtel.org/v1/webapi/oidc/callback` |
| `tctl must be used on an Auth Service host` | Убедитесь, что Teleport запущен и порт 3025 слушается |
| `Пароль запрашивается, хотя есть .pgpass` | Проверьте права: `chmod 600 ~/.pgpass`. Убедитесь, что порт в файле совпадает с подключаемым |
| `.pgpass` игнорируется при подключении к HAProxy | Добавьте строку с портом `5000`, а не `5432` |

## 11. Работа с PostgreSQL через psql (справочник)

### Подключение к БД

```bash
# Подключение к HAProxy (балансировщик)
psql -h 192.168.45.201 -p 5000 -U teleport_user -d teleport

# Подключение напрямую к мастеру
psql -h 192.168.45.201 -p 5432 -U teleport_user -d teleport

# Подключение от суперпользователя
psql -h 192.168.45.201 -p 5000 -U postgres
```

### Основные команды psql

| Команда | Описание |
|---------|----------|
| `\l` | Список всех баз данных |
| `\dt` | Список таблиц в текущей БД |
| `\du` | Список пользователей/ролей |
| `\c teleport` | Переключиться на БД teleport |
| `\d kv` | Описание структуры таблицы kv |
| `\x` | Включить расширенный вывод (построчный) |
| `\q` | Выйти из psql |
| `\?` | Справка по командам |
| `\h` | Справка по SQL командам |

### Просмотр данных Teleport

```bash
# Просмотр ключей пользователей
SELECT key FROM kv WHERE key LIKE '%user%' LIMIT 10;

# Просмотр всех ключей
SELECT key FROM kv LIMIT 20;

# Подсчет количества записей
SELECT COUNT(*) FROM kv;

# Просмотр значения по ключу (в сыром виде)
SELECT value FROM kv WHERE key = '/web/users/testuser/params';
```
