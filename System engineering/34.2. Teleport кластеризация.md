# Teleport Enterprise v16: Полное руководство по развертыванию HA-кластера

**Актуальная архитектура после опытной эксплуатации:**
- **teleport1** — Auth + Proxy + App (Master)
- **teleport2** — Proxy только (Standby)
- **teleport3** — Auth только (Witness для кворума Keepalived)
- **PostgreSQL (Patroni + HAProxy)** — общее хранилище состояния

В данном руководстве описан процесс настройки отказоустойчивого кластера Teleport с использованием внешней БД PostgreSQL, балансировщика HAProxy, Keepalived и интеграции с Keycloak (OIDC).

## 2. Настройка базы данных PostgreSQL

Teleport v16 требует логической репликации для синхронизации кэша между нодами.

### 2.1. Установка wal2json (обязательно!)

```bash
# На всех PostgreSQL серверах
apt install postgresql-15-wal2json
systemctl restart postgresql
```

### 2.2. Создание БД и пользователя

Зайдите на мастер-ноду PostgreSQL и выполните:

```bash
sudo -u postgres psql
```

```sql
-- Создаем пользователя
CREATE USER teleport_user WITH PASSWORD 'VeryStrong77!';

-- Создаем базу данных
CREATE DATABASE teleport OWNER teleport_user;

-- Даем права
GRANT ALL PRIVILEGES ON DATABASE teleport TO teleport_user;

-- ВАЖНО: Для работы Replication Slots (Change Feed) в v16 Enterprise
-- пользователю требуются права SUPERUSER или REPLICATION
ALTER USER teleport_user WITH REPLICATION;

-- Проверяем созданных пользователей
\du
```

**Вывод команды `\du`:**

```
                                             Список ролей
   Имя роли    |                                Атрибуты                                 | Член ролей 
---------------+-------------------------------------------------------------------------+------------
 jira          |                                                                         | {}
 keycloak      |                                                                         | {}
 postgres      | Суперпользователь, Создаёт роли, Создаёт БД, Репликация, Пропускать RLS | {}
 replicator    | Репликация                                                              | {}
 rt_pbx        |                                                                         | {}

```

### 2.3. Настройка .pgpass для автоматизации

Чтобы утилиты могли подключаться к БД без ввода пароля:

```bash
# На всех серверах (teleport1, teleport2, teleport3, pg1)
cat >> ~/.pgpass <<EOF
192.168.45.201:5000:teleport:teleport_user:VeryStrong77!
EOF
chmod 600 ~/.pgpass
```

### 2.4. Установка прав на файл .pgpass

Права должны быть строго `600`, иначе PostgreSQL игнорирует файл:

```bash
chmod 600 ~/.pgpass
```

## 6. Балансировка через HAProxy

### 6.1. Конфигурация HAProxy для PostgreSQL и Teleport

```bash
mcedit /etc/haproxy/haproxy.cfg
```

Добавить к существующей конфигурации с pg кластером:

```haproxy
## PostgreSQL через HAProxy
frontend postgres_frontend
    bind *:5000
    mode tcp
    default_backend postgres_master

backend postgres_master
    mode tcp
    balance roundrobin
    option httpchk GET /master
    server pg1 192.168.45.201:5432 check port 8008 inter 5s rise 2 fall 3
    server pg2 192.168.45.202:5432 check port 8008 inter 5s rise 2 fall 3
    server pg3 192.168.45.204:5432 check port 8008 inter 5s rise 2 fall 3

## TELEPORT
frontend teleport-web
    bind *:443
    mode tcp
    default_backend teleport-proxy-web

frontend teleport-ssh
    bind *:3023
    mode tcp
    default_backend teleport-proxy-ssh

frontend teleport-tunnel
    bind *:3080
    mode tcp
    default_backend teleport-proxy-tunnel

backend teleport-proxy-web
    mode tcp
    balance source
    option tcp-check
    server teleport-vip 192.168.87.240:443 check

backend teleport-proxy-ssh
    mode tcp
    balance source
    option tcp-check
    server teleport-vip 192.168.87.240:3023 check

backend teleport-proxy-tunnel
    mode tcp
    balance source
    option tcp-check
    server teleport-vip 192.168.87.240:3080 check
```

### 6.2. Запуск HAProxy

```bash
haproxy -c -f /etc/haproxy/haproxy.cfg
systemctl restart haproxy
```

Статистика по адресу `http://192.168.45.201:7000/`

## 10. Типичные проблемы и решения

| Проблема | Решение |
|----------|---------|
| `field oidc not found in type config.Auth` | В v16 OIDC коннектор создаётся через отдельный файл, не в `teleport.yaml` |
| `must be superuser or replication role` | `ALTER USER teleport_user WITH REPLICATION;` |
| `could not access file "wal2json"` | Установите пакет `postgresql-15-wal2json` |
| `second_factor cannot be off` | Установите `second_factor: on` и настройте `webauthn` |
| `listen tcp 192.168.87.238:3025: bind: cannot assign requested address` | Исправьте `listen_addr` и `public_addr` на IP текущего сервера |
| `redirect_uri mismatch` | Добавьте в Keycloak точный URL: `https://teleport.runtel.org/v1/webapi/oidc/callback` |
| `tctl must be used on an Auth Service host` | Выполняйте на узле с `auth_service: yes` (teleport1 или teleport3) |
| `Пароль запрашивается, хотя есть .pgpass` | `chmod 600 ~/.pgpass`, порт должен совпадать |
| `.pgpass` игнорируется при подключении к HAProxy | Добавьте строку с портом `5000`, а не `5432` |
| `token does not allow "Proxy" role` | `tctl nodes add --roles=proxy` |
| `app_service требует proxy_server вместо auth_server` | На teleport2 выключи `app_service` |

## 12. Почему два сервера НЕ могут быть активным Auth (ВАЖНО)

В процессе настройки выяснилось, что попытка сделать оба сервера (`teleport1` и `teleport2`) активными `Auth`-серверами ведёт к нестабильности и ошибкам.

**Причины:**

1.  **Split-Brain (разделённое сознание)**
    - Оба сервера пытаются управлять одними и теми же ресурсами в PostgreSQL.
    - Возникает состояние гонки (race condition), зависание на этапе инициализации.

2.  **Конфликт сертификатов**
    - Центр сертификации (CA) должен быть единственным.
    - Два активных Auth могут расходиться в подписи сертификатов.

3.  **Логическая репликация в PostgreSQL**
    - Teleport использует механизм Change Feed, который опирается на **единственную активную подписку (Replication Slot)**.
    - Второй Auth конфликтует за этот слот.

**Правильная архитектура HA для Teleport — Active‑Standby:**
- **Один активный Auth** (teleport1)
- **Второй узел — только Proxy** (teleport2)
- **Третий узел — Witness Auth для кворума Keepalived** (teleport3)

## 13. Добавление третьего узла (teleport3) только для кворума Auth

**Зачем нужен teleport3?**
- Обеспечивает кворум в Keepalived (3 узла вместо 2).
- Никогда не захватывает VIP (низкий приоритет 100).
- Может быть использован как резервный Auth при ручном переключении.
- Потребляет минимум ресурсов (1 vCPU, 2GB RAM).

**Конфигурация teleport3 описана в разделе 3.3.**

**Порядок запуска кластера из трёх узлов:**
```bash
# 1. Очистить все узлы
systemctl stop teleport
rm -rf /var/lib/teleport/proc/* /var/lib/teleport/backend/*

# 2. Запустить teleport1 (Auth + Proxy + App)
systemctl start teleport

# 3. Запустить teleport3 (Auth только)
systemctl start teleport

# 4. Запустить teleport2 (Proxy только)
systemctl start teleport

# 5. Запустить Keepalived на всех трёх узлах
systemctl restart keepalived
```

**Проверка:**
```bash
ip a s | grep 192.168.87.240   # VIP только на teleport1
tctl status                     # показывает кластер
tctl get nodes
```
