# Полное руководство по развертыванию HA-кластера Teleport Enterprise v16

В данном руководстве описан процесс настройки отказоустойчивого кластера Teleport с использованием внешней БД PostgreSQL, балансировщика HAProxy и интеграции с Keycloak (OIDC).

## 1. Подготовка инфраструктуры

| Узел | IP-адрес | Роль |
|-----------|----------|------|
| **teleport1** | 192.168.87.238 | Auth + Proxy + App |
| **teleport2** | 192.168.87.239 | Auth + Proxy + App |
| **pg1 (Leader)** | 192.168.45.201 | PostgreSQL (порт 5000 через HAProxy) |

---

## 2. Настройка базы данных PostgreSQL

Teleport v16 требует логической репликации для синхронизации кэша между нодами.

### 2.1. Создание БД и пользователя
Зайдите на мастер-ноду PostgreSQL и выполните:

```sql
-- Создаем пользователя
CREATE USER teleport_user WITH PASSWORD 'TeleportPass89!';

-- Создаем базу данных
CREATE DATABASE teleport OWNER teleport_user;

-- Даем права
GRANT ALL PRIVILEGES ON DATABASE teleport TO teleport_user;

-- ВАЖНО: Для работы Replication Slots (Change Feed) в v16 Enterprise
-- пользователю требуются права SUPERUSER.
ALTER USER teleport_user WITH SUPERUSER;
```

### 2.2. Настройка .pgpass для автоматизации
Чтобы `tctl` и другие утилиты могли подключаться к БД без ввода пароля, создайте файл `~/.pgpass`:

```bash
# Формат: хост:порт:база:пользователь:пароль
echo "192.168.45.201:5000:teleport:teleport_user:TeleportPass89!" > ~/.pgpass
chmod 0600 ~/.pgpass
```

## 3. Настройка первого узла (teleport1)

### 3.1. Редактирование /etc/teleport.yaml
Мы столкнулись с тем, что v16 Enterprise требует строгого соблюдения правил MFA.

```bash
mcedit /etc/teleport.yaml
```

**Содержимое файла:**
```yaml
version: v3
teleport:
  nodename: teleport1.runtel.org
  advertise_ip: 192.168.87.238
  data_dir: /var/lib/teleport
  storage:
    type: postgres
    conn_string: "postgres://teleport_user:TeleportPass89!@192.168.45.201:5000/teleport?sslmode=disable"
  auth_server: 192.168.87.238:3025

auth_service:
  enabled: yes
  license_file: /var/lib/teleport/license.pem
  listen_addr: 0.0.0.0:3025
  public_addr: 192.168.87.238:3025
  proxy_listener_mode: multiplex
  authentication:
    type: oidc
    second_factor: on             # В v16 Enterprise нельзя ставить off
    webauthn:
      rp_id: teleport.runtel.org  # Домен вашего прокси
  cluster_name: teleport-runtel

proxy_service:
  enabled: yes
  public_addr: teleport.runtel.org:443
  https_keypairs:
    - cert_file: /etc/letsencrypt/live/teleport.runtel.org/fullchain.pem
      key_file: /etc/letsencrypt/live/teleport.runtel.org/privkey.pem
```

### 3.2. Запуск и проверка
```bash
systemctl restart teleport
ss -tlnp | grep 3025  # Убеждаемся, что порт открыт
```

## 4. Настройка OIDC (Keycloak)

В v16 коннекторы нельзя описывать в `teleport.yaml`. Их нужно создавать как динамические ресурсы.

### 4.1. Создание манифеста коннектора
Создайте директорию и файл:
```bash
mkdir -p /etc/teleport.d/
mcedit /etc/teleport.d/keycloak-connector.yaml
```

**Содержимое `keycloak-connector.yaml`:**
```yaml
kind: oidc
version: v3
metadata:
  name: keycloak
spec:
  client_id: teleport
  client_secret: "HgO2LUuSb7zKvtxgL2sG5aEErjp1ddop"
  issuer_url: "https://sso.runtel.ru:8443/realms/runtel"
  redirect_url: "https://teleport.runtel.org/v1/webapi/oidc/callback"
  scope: ["openid", "profile", "email", "groups"]
  claims_to_roles:
    # Маппинг администраторов
    - claim: groups
      value: jira-administrators
      roles: ["access", "editor"]
    - claim: email
      value: "k@runtel.ru"
      roles: ["access", "editor"]
    # Маппинг инженеров
    - claim: groups
      value: engineers
      roles: ["access"]
```

### 4.2. Применение коннектора
```bash
tctl create -f /etc/teleport.d/keycloak-connector.yaml
```

Второй узел настраивается аналогично первому, но с уникальным именем и IP. Коннектор Keycloak он подтянет из Postgres автоматически.

## 6. Балансировка через HAProxy

Чтобы оба прокси работали под одним доменом, настройте HAProxy:

```haproxy
frontend teleport-https
    bind *:443
    mode tcp
    default_backend teleport-proxies

backend teleport-proxies
    mode tcp
    balance roundrobin
    server teleport1 192.168.87.238:443 check
    server teleport2 192.168.87.239:443 check
```

## 7. Диагностика (Troubleshooting)

### 7.1. Проверка логов OIDC
Если логин не срабатывает, смотрите логи в реальном времени:
```bash
journalctl -u teleport -f
```
Ищите секцию `OIDC ID Token claims`. Там будет список групп, которые прислал Keycloak. Если вашей группы нет в списке — проверьте Mappers в Keycloak.

### 7.2. Проверка синхронизации
Выполните на **teleport2**:
```bash
tctl get users
```
Если вы видите там пользователя, который заходил через первый сервер — кластер работает идеально.
