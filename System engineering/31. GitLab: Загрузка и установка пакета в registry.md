## Часть 3. Проблемы при загрузке и их решение

### 3.1. Проблема: ошибка 401 Unauthorized

**Симптом:**
```
INFO     Response: 401 Unauthorized
{"message":"401 Unauthorized"}
```

**Причина:** Использовался невалидный или просроченный токен.

**Решение:** Создали новый токен с правами `api` и `write_repository`, проверили его работу через `curl`.

---

### 3.2. Проблема: ошибка 404 при загрузке в группу

**Симптом:**
```
twine upload --repository-url https://gitlab.runtel.org/api/v4/groups/10/-/packages/pypi ...
INFO     Response: 404 Not Found
```

**Причина:** GitLab не поддерживает загрузку пакетов через групповой endpoint (только установку).

**Решение:** Использовать endpoint с `project_id`: `/api/v4/projects/113/packages/pypi`

---

### 3.3. Проблема: ошибка 400 при повторной загрузке

**Симптом:**
```
INFO     Response: 400 Bad Request
{"message":"400 Bad request - Validation failed: File name has already been taken"}
```

**Причина:** Пакет с таким именем и версией уже существует в репозитории.

**Решение:** Загружать только новые версии пакетов или исключать уже существующие из списка загрузки.

---

### 3.4. Проблема: ошибка с метаданными pytest-9.0.2

**Симптом:**
```
ERROR InvalidDistribution: Invalid distribution metadata: license-expression introduced in metadata version 2.4, not 2.1
```

**Причина:** `pytest` версии 9.0.2 использует формат метаданных 2.4, который содержит поле `license-expression`. Инструмент `twine` или GitLab PyPI API не поддерживает эту версию метаданных.

**Решение:** Загрузить более старую версию `pytest` (8.3.4), которая использует метаданные версии 2.1.

```bash
# Скачать pytest 8.3.4
pip download pytest==8.3.4 --dest /tmp/packages --no-deps

# Загрузить его
twine upload --repository gitlab pytest-8.3.4-py3-none-any.whl
```

# Загрузка и установка пакетов из внутреннего GitLab проекта

## Способ 1: Указать index-url в команде (рекомендуется для разовых установок)

```bash
pip install -r requirements.txt \
  --index-url https://gitlab.runtel.org/api/v4/projects/113/packages/pypi/simple \
  --no-index
```

**Важно:** `--no-index` запрещает pip обращаться к PyPI вообще.

## Способ 2: Настроить pip.conf (для постоянного использования)

Создайте или отредактируйте файл `~/.pip/pip.conf`:

```ini
[global]
index-url = https://gitlab.runtel.org/api/v4/projects/113/packages/pypi/simple
no-index = true
```

Или если нужно иметь возможность использовать PyPI как запасной источник:

```ini
[global]
index-url = https://gitlab.runtel.org/api/v4/projects/113/packages/pypi/simple
extra-index-url = https://pypi.org/simple
```

После этого `pip install -r requirements.txt` будет использовать GitLab как основной источник.

## Способ 3: Использовать переменные окружения

```bash
export PIP_INDEX_URL="https://gitlab.runtel.org/api/v4/projects/113/packages/pypi/simple"
export PIP_NO_INDEX="true"
pip install -r requirements.txt
```

## Способ 4: Создать requirements.txt с указанием индекса (нестандартно)

В самом `requirements.txt` можно указать индекс, добавив в начало файла:

```txt
--index-url https://gitlab.runtel.org/api/v4/projects/113/packages/pypi/simple
--no-index

allure-pytest==2.14.2
allure-python-commons==2.14.2
...
```

Но этот способ менее гибкий, так как параметры в requirements.txt могут быть не поддержаны в некоторых средах.

---

## Важное замечание по версиям в вашем requirements.txt

Обратите внимание на версии:

1. **pytest==9.0.2** — эта версия не проиндексировалась в GitLab (метаданные 2.4). В репозитории доступна только `pytest==8.3.4`. Если вы укажете `pytest==9.0.2`, pip не сможет её найти.

2. **pytest-base-url==2.1.0** — эта версия тоже вызвала ошибку с метаданными. Возможно, она не загрузилась.

3. **runtel_logger** — без версии. Убедитесь, что в GitLab есть этот пакет.

# Инструкция: Загрузка бинарных файлов в GitLab Generic Package Registry

### Что мы сделали
Мы загрузили ZIP-архив с программой `dysk` в GitLab как **Generic Package**. Это позволяет хранить любые файлы (бинарники, архивы, конфиги) в системе управления пакетами GitLab, не оборачивая их в Python-пакеты.

### 1. Загрузка файла

```bash
# Базовый синтаксис
curl --request PUT \
     --header "PRIVATE-TOKEN: ваш_токен" \
     --upload-file локальный_файл \
     "https://gitlab.gitlab.org/api/v4/projects/ID_проекта/packages/generic/имя_пакета/версия/имя_файла"
```

**Конкретный пример:**
```bash
curl --request PUT \
     --header "PRIVATE-TOKEN: glpat-fVfyK2aSL23hnCZaEsSA5W86MQp1OjFqCA.01.0y03yeb0c" \
     --upload-file dysk_3.6.0.zip \
```

### 6. Что можно хранить в Generic Packages

- **Любые бинарные файлы** — программы, библиотеки
- **Архивы** — .zip, .tar.gz, .7z
- **Документацию** — PDF, HTML
- **Конфигурации** — JSON, YAML
- **Образы Docker** (хотя лучше использовать Container Registry)
- **Любые артефакты сборки**

### 7. Особенности и ограничения

- **Имена:** Только буквы (a-z, A-Z), цифры, точки, дефисы, подчеркивания
- **Максимальный размер файла:** Зависит от настроек GitLab (обычно до 5-10 ГБ)
- **Версионирование:** Можно загружать несколько версий одного пакета
- **Права доступа:** Наследуются от прав на проект

### 8. Автоматизация с CI/CD

В `.gitlab-ci.yml` можно добавить загрузку артефактов:

```yaml
upload-package:
  stage: deploy
  script:
    - curl --request PUT
        --header "PRIVATE-TOKEN: $GITLAB_TOKEN"
        --upload-file dysk_3.6.0.zip
        "https://gitlab.runtel.org/api/v4/projects/$CI_PROJECT_ID/packages/generic/dysk/3.6.0/dysk_3.6.0.zip"
  only:
    - tags
```

### Заключение

Generic Package Registry в GitLab — это простой способ хранить любые файлы с версионированием, доступом через API и веб-интерфейс. Не требует специальной подготовки файлов и идеально подходит для бинарных артефактов, архивов и других ресурсов, не связанных с конкретными языками программирования.

# Дополнение: Загрузка npm-пакетов в GitLab Package Registry

## Аналогия с PyPI

| Действие | PyPI | npm |
|----------|------|-----|
| Скачивание пакета | `pip download` | `npm pack` или `wget` из registry |
| Файл пакета | `.whl` или `.tar.gz` | `.tgz` |
| Конфиг аутентификации | `.pypirc` | `.npmrc` |
| Загрузка в GitLab | `twine upload` | `npm publish` |
| Установка из GitLab | `pip install --index-url` | `npm install --registry` |
| Полная изоляция | `--no-index` | `--no-default-registry` |

## Часть 0. Получение npm-пакета из публичного registry

### 0.1. Скачивание конкретной версии пакета

```bash
# Создаем временную директорию
mkdir /tmp/npm_packages && cd /tmp/npm_packages

# Способ 1: через npm pack (требует npm)
npm pack glob@7.2.3

# Способ 2: через wget (прямая ссылка на файл)
wget https://registry.npmjs.org/glob/-/glob-7.2.3.tgz
wget https://registry.npmjs.org/glob/-/glob-13.0.6.tgz
```

**Результат:**
```
glob-7.2.3.tgz  (16 KB)
glob-13.0.6.tgz (365 KB)
```

### 0.2. Скачивание пакета со всеми зависимостями

Для полной изоляции (как в случае с PyPI) нужно скачать все зависимости:

```bash
# Создаем пустой проект
mkdir offline-project && cd offline-project
npm init -y

# Устанавливаем пакет (без установки зависимостей в систему)
npm install glob@7.2.3 --package-lock-only

# Анализируем зависимости из package-lock.json
cat package-lock.json | grep -A2 '"requires"' | grep -v "^--$"

# Скачиваем все зависимости в папку
npm pack glob@7.2.3
for pkg in $(npm list --depth=0 --parseable --prod); do
  if [ -d "$pkg" ] && [ "$pkg" != "." ]; then
    (cd "$pkg" && npm pack)
  fi
done
```

## Часть 1. Подготовка и загрузка npm-пакета в GitLab

### 1.1. Создание структуры для загрузки

```bash
cd /tmp/npm_packages

# Создаем package.json (обязателен для публикации)
cat > package.json << 'EOF'
{
  "name": "glob",
  "version": "7.2.3",
  "description": "Glob package - offline distribution"
}
EOF
```

### 1.2. Настройка аутентификации (.npmrc)

Аналог файла `.pypirc` для PyPI:

```bash
cat > .npmrc << 'EOF'
//gitlab.runtel.org/api/v4/projects/ID_ПРОЕКТА/packages/npm/:_authToken=ВАШ_ТОКЕН
EOF
```
<!-- authToken=glpat-fVfyK2aSL23hnCZaEsSA5W86MQp1OjFqCA.01.0y03yeb0c -->


**Важно:** 
- Используется формат `//URL/путь/:_authToken`
- Токен должен иметь права `api` и `write_repository` (как для PyPI)

### 1.3. Загрузка пакета в GitLab

```bash
# Загружаем версию 7.2.3
npm publish glob-7.2.3.tgz \
  --registry=https://gitlab.runtel.org/api/v4/projects/45/packages/npm/

# Меняем версию в package.json
sed -i 's/"version": "7.2.3"/"version": "13.0.6"/' package.json

# Загружаем версию 13.0.6
npm publish glob-13.0.6.tgz \
  --registry=https://gitlab.runtel.org/api/v4/projects/45/packages/npm/
```

**Успешный ответ:**
```
+ glob@7.2.3
+ glob@13.0.6
```

### 1.4. Альтернативная загрузка через curl (без npm)

Если npm недоступен, можно использовать прямой API:

```bash
curl --request PUT \
     --header "PRIVATE-TOKEN: ВАШ_ТОКЕН" \
     --header "Content-Type: application/octet-stream" \
     --data-binary @glob-7.2.3.tgz \
     "https://gitlab.runtel.org/api/v4/projects/45/packages/npm/glob/-/glob-7.2.3.tgz"
```

## Часть 3. Проблемы и их решение

### 3.1. Ошибка: Package already exists

**Симптом:**
```
npm ERR! 400 Bad Request - PUT https://gitlab.../glob/-/glob-7.2.3.tgz
```

**Причина:** Пакет с такой версией уже существует.

**Решение:** 
- Загружать только новые версии
- Или удалить существующий пакет через API:
  ```bash
  curl --request DELETE \
       --header "PRIVATE-TOKEN: ВАШ_ТОКЕН" \
       "https://gitlab.runtel.org/api/v4/projects/45/packages/272"
  ```

### 3.2. Ошибка: 401 Unauthorized

**Симптом:**
```
npm ERR! code E401
npm ERR! Unable to authenticate
```

**Причина:** Невалидный или просроченный токен.

**Решение:** Создать новый токен с правами `api` и `write_repository`.

### 3.3. Ошибка: Неправильный .npmrc формат

**Неправильно:**
```ini
registry=https://gitlab.runtel.org/api/v4/projects/45/packages/npm/
_authToken=glpat-xxx
```

**Правильно:**
```ini
//gitlab.runtel.org/api/v4/projects/45/packages/npm/:_authToken=glpat-xxx
```

## Часть 4. Установка пакетов из GitLab

### 4.1. Временная установка (с указанием registry)

```bash
# Установка конкретной версии
npm install glob@7.2.3 \
  --registry=https://gitlab.runtel.org/api/v4/projects/45/packages/npm/

# Установка последней версии
npm install glob \
  --registry=https://gitlab.runtel.org/api/v4/projects/45/packages/npm/
```

### 4.2. Постоянная настройка через .npmrc

Создайте `.npmrc` в корне проекта:

```ini
registry=https://gitlab.runtel.org/api/v4/projects/45/packages/npm/
//gitlab.runtel.org/api/v4/projects/45/packages/npm/:_authToken=glpat-fVfyK2aSL23hnCZaEsSA5W86MQp1OjFqCA.01.0y03yeb0c
```

После этого можно устанавливать просто:
```bash
npm install glob@7.2.3
```

### 4.3. Полная изоляция от публичного npm

Как и в случае с PyPI (`--no-index`), для npm используется `--no-default-registry`:

```bash
npm install glob@7.2.3 \
  --registry=https://gitlab.runtel.org/api/v4/projects/45/packages/npm/ \
  --no-default-registry
```

**Важно:** При использовании `--no-default-registry` в вашем GitLab реестре должны быть **все зависимости**, иначе установка не удастся.

### 4.4. Установка через переменные окружения

```bash
export npm_config_registry="https://gitlab.runtel.org/api/v4/projects/45/packages/npm/"
export npm_config_//gitlab.runtel.org/api/v4/projects/45/packages/npm/:_authToken="glpat-xxx"
npm install glob@7.2.3
```

## Часть 7. Пример полного рабочего скрипта

```bash
#!/bin/bash
# upload-npm-package.sh - автоматическая загрузка npm пакетов в GitLab

TOKEN="glpat-fVfyK2aSL23hnCZaEsSA5W86MQp1OjFqCA.01.0y03yeb0c"
PROJECT_ID="45"
PACKAGE_NAME="glob"
VERSIONS=("7.2.3" "13.0.6")

cd /tmp/npm_packages || exit 1

# Скачиваем пакеты
for VERSION in "${VERSIONS[@]}"; do
    if [ ! -f "${PACKAGE_NAME}-${VERSION}.tgz" ]; then
        echo "Downloading ${PACKAGE_NAME}@${VERSION}"
        npm pack "${PACKAGE_NAME}@${VERSION}"
    fi
done

# Создаем .npmrc
cat > .npmrc << EOF
//gitlab.runtel.org/api/v4/projects/${PROJECT_ID}/packages/npm/:_authToken=${TOKEN}
EOF

# Загружаем каждую версию
for VERSION in "${VERSIONS[@]}"; do
    echo "Publishing ${PACKAGE_NAME}@${VERSION}"
    
    # Создаем package.json с нужной версией
    cat > package.json << EOF
{
  "name": "${PACKAGE_NAME}",
  "version": "${VERSION}",
  "description": "${PACKAGE_NAME} package - offline distribution"
}
EOF
    
    npm publish "${PACKAGE_NAME}-${VERSION}.tgz" \
        --registry="https://gitlab.runtel.org/api/v4/projects/${PROJECT_ID}/packages/npm/"
done

echo "All packages published successfully!"

# Проверяем
echo "Checking published packages:"
curl --silent --header "PRIVATE-TOKEN: ${TOKEN}" \
    "https://gitlab.runtel.org/api/v4/projects/${PROJECT_ID}/packages?package_name=${PACKAGE_NAME}" | \
    python3 -c "import sys, json; [print(f\"{p['name']} {p['version']}\") for p in json.load(sys.stdin)]"
```

## Заключение

GitLab Package Registry поддерживает npm-пакеты нативно, что позволяет:
1. Хранить npm-пакеты в корпоративном GitLab
2. Управлять версиями через стандартные npm команды
3. Контролировать доступ через токены и права проекта
4. Использовать в CI/CD без доступа к публичному npm

В отличие от PyPI, у npm нет проблем с версиями метаданных (как с `pytest-9.0.2`), но есть свои особенности с аутентификацией и форматом `.npmrc`.
