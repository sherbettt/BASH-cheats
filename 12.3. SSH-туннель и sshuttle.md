# Организация SSH-туннелей и VPN-подобного доступа через bastion-хост

В этой статье рассмотрим практические примеры организации безопасного доступа к внутренним ресурсам сети через роутер с Ubuntu. Используем SSH-туннели (local port forwarding) и утилиту `sshuttle` для прозрачного проксирования трафика.

> **Полезные ресурсы:**
> - [A visual guide to SSH tunnels](https://robotmoon.com/ssh-tunnels/) — наглядное объяснение работы туннелей
> - [Настройка NAT на роутере](https://github.com/sherbettt/BASH-cheats/blob/main/26.%20Настройка%20NAT%20на%20роутере%20с%20Ubuntu.md) — предыдущая часть
> - [man sshuttle](https://sshuttle.readthedocs.io/en/stable/manpage.html) — официальная документация

---

## Схема сети

```
Internet ────── Router (Ubuntu) ─────┐
                eth0: 192.168.87.112/24 (внешний, шлюз 192.168.87.1)
                eth2: 192.168.96.113/24 (внешний, шлюз 192.168.96.1)
                      │
                      │ eth1: 192.168.56.1/24 (внутренний, без выхода в интернет)
                      ↓
         ┌───────────────────────┐
         │  Internal Network     │
         │  192.168.56.0/24      │
         │                       │
         │  Host1: 192.168.56.2  │
         │  Host2: 192.168.56.3  │
         └───────────────────────┘
```

**Участники:**
- **Роутер (Ubuntu)**: `192.168.87.112` (внешний IP), `192.168.56.1` (внутренний)
- **Клиентская машина**: `192.168.87.136` (Ximper Linux) — с неё будем устанавливать соединения
- **Внутренние хосты**: `192.168.56.2` и `192.168.56.3` (без прямого доступа в интернет)

**Важно:** SSH-ключи от машины `192.168.87.136` должны быть предварительно скопированы на роутер (`192.168.87.112`).

## Подготовка окружения

### 1. Установка необходимых пакетов

На всех машинах, которые будут участвовать в соединениях, установите OpenSSH-сервер:

```bash
sudo apt update && sudo apt install openssh-server -y
```

На клиентской машине (`192.168.87.136`) установите `sshuttle`:

```bash
# Способ 1 — через apt (рекомендуемый)
sudo apt install sshuttle -y

# Способ 2 — через pip
sudo pip install sshuttle

# Способ 3 — из исходников
git clone https://github.com/sshuttle/sshuttle.git
cd sshuttle
sudo ./setup.py install
```

### 2. Настройка firewall на клиенте (`192.168.87.136`)

Если используется `firewalld`:

```bash
# Установка и запуск
sudo apt install firewalld -y
sudo systemctl start firewalld
sudo systemctl enable firewalld

# Открываем порт 2222 для SSH-туннеля
sudo firewall-cmd --zone=public --add-port=2222/tcp --permanent
sudo firewall-cmd --reload

# Проверка правил
sudo firewall-cmd --list-all
```

Если используется `iptables`:

```bash
# Разрешаем входящие соединения на порт 2222
sudo iptables -A INPUT -p tcp --dport 2222 -j ACCEPT

# Сохраняем правила (для Debian/Ubuntu)
sudo iptables-save > /etc/iptables/rules.v4
# Или для систем с service iptables
sudo service iptables save
```

**Важно:** Для корректной работы `sshuttle` на клиенте может потребоваться:
- Включенный IP-форвардинг: `sudo sysctl -w net.ipv4.ip_forward=1`
- Разрешённый в firewall трафик для создаваемых туннелей

## Решение №1: Local Port Forwarding (SSH-туннель)

**Цель:** Создать SSH-туннель от клиента (`192.168.87.136`) до внутреннего хоста (`192.168.56.2`) через роутер.

### Базовый туннель

На клиенте выполняем:

```bash
ssh -L 2222:192.168.56.2:22 root@192.168.87.112
```

**Что происходит:**
- **`-L 2222:192.168.56.2:22`** — локальный портфорвардинг: все соединения к порту `2222` на клиенте перенаправляются на порт `22` (SSH) хоста `192.168.56.2`
- **`root@192.168.87.112`** — подключение к роутеру, который будет проксировать трафик

**Расшифровка параметров:**
```
-L [локальный_IP:]локальный_порт:целевой_IP:целевой_порт пользователь@шлюз
```

**Другие варианты:**

```bash
# Явное указание интерфейса для прослушивания
ssh -L 127.0.0.1:2222:192.168.56.2:22 root@192.168.87.112

# Проброс HTTP-трафика (порт 80)
ssh -L 8080:192.168.56.2:80 root@192.168.87.112

# Проброс всех интерфейсов (внимание: безопасность!)
ssh -L 0.0.0.0:2222:192.168.56.2:22 root@192.168.87.112
```

### Подключение через туннель

В другой сессии терминала (пока туннель активен):

```bash
ssh root@localhost -p 2222
```

Или если нужно скопировать файлы:

```bash
scp -P 2222 file.txt root@localhost:/tmp/
```

### Продвинутые опции для стабильного туннеля

Чтобы туннель не обрывался и работал в фоне:

```bash
ssh -fN -L 2222:192.168.56.2:22 root@192.168.87.112
```

- **`-f`** — уйти в фон после аутентификации
- **`-N`** — не выполнять удалённые команды (только туннель)
- **`-T`** — отключить псевдо-TTY

Для автоматического переподключения используйте `autossh`:

```bash
autossh -M 0 -o "ServerAliveInterval 60" -o "ServerAliveCountMax 3" \
        -L 2222:192.168.56.2:22 root@192.168.87.112
```

## Решение №2: Использование sshuttle (прозрачный VPN)

**Цель:** Создать VPN-подобное соединение, чтобы весь трафик к подсети `192.168.56.0/24` прозрачно проксировался через роутер.

### Базовый вариант

```bash
sudo sshuttle -vv -r root@192.168.87.112 192.168.56.0/24
```

**Расшифровка:**
- **`sudo`** — требуется для изменения таблиц маршрутизации и iptables
- **`-vv`** — подробный вывод (для отладки; можно убрать в production)
- **`-r root@192.168.87.112`** — удалённый сервер (роутер)
- **`192.168.56.0/24`** — подсеть, трафик к которой будет направляться через туннель

### С указанием конкретного SSH-ключа

```bash
sudo sshuttle -r root@192.168.87.112 192.168.56.0/24 \
              --ssh-cmd "ssh -i /home/kirill/.ssh/id_ed25519"
```

### С пробросом DNS

```bash
sudo sshuttle --dns -r root@192.168.87.112 192.168.56.0/24
```

Опция `--dns` перехватывает DNS-запросы и перенаправляет их на удалённую сторону, что полезно для доступа к внутренним доменным именам.

### Проксирование всего трафика (кроме исключений)

```bash
# Весь IPv4-трафик через туннель, исключая сам роутер
sudo sshuttle -r root@192.168.87.112 0/0 -x 192.168.87.112/32
```

### Использование с автоматическим определением сетей

```bash
sudo sshuttle -vNHr root@192.168.87.112 -x 192.168.87.112:22
```

- **`-N`** — автоматически определить маршруты на сервере
- **`-H`** — обновлять `/etc/hosts` для обнаруженных хостов

### Работа с firewalld

Если используется `firewalld`, на клиенте нужно разрешить трафик для интерфейса, который использует `sshuttle`:

```bash
sudo firewall-cmd --permanent --add-rich-rule='rule family="ipv4" forward-port port="0" protocol="tcp" to-port="0" accept'
sudo firewall-cmd --reload
```

Или временно отключить firewall для тестирования:
```bash
sudo systemctl stop firewalld
```

### Проверка подключения

После запуска `sshuttle` должно быть возможно прямое подключение к внутренним хостам:

```bash
ssh root@192.168.56.2
ping 192.168.56.3
curl http://192.168.56.2:80
```

## Решение №3: SSH как ProxyCommand (bastion host)

**Цель:** Использовать роутер в качестве прыжкового хоста (jump host) для прямого подключения к внутренним машинам.

### Вариант с ProxyCommand

```bash
ssh -o ProxyCommand="ssh -W %h:%p root@192.168.87.112" root@192.168.56.2
```

**Как это работает:**
- `ssh -W %h:%p root@192.168.87.112` — через роутер устанавливается соединение к целевому хосту (`%h` — хост, `%p` — порт)
- `root@192.168.56.2` — финальное подключение к внутреннему хосту

### Вариант с Jumphost (более современный)

```bash
ssh -J root@192.168.87.112 root@192.168.56.2
```

**Для цепочек хостов:**
```bash
ssh -J user1@host1,user2@host2 root@192.168.56.2
```

### Создание туннеля через bastion

```bash
ssh -J root@192.168.87.112 -L 80:192.168.56.2:80 root@192.168.56.2
```

Теперь можно открыть в браузере `http://localhost:80` и увидеть веб-интерфейс внутреннего хоста.

## Дополнительные важные моменты

### 1. Проброс привилегированных портов (1-1023)

Для работы с портами ниже 1024 требуется запуск с `sudo`:

```bash
sudo ssh -L 443:192.168.56.2:443 root@192.168.87.112
```

### 2. Глобальный доступ к туннелю (GatewayPorts)

Чтобы туннель был доступен не только с localhost:

```bash
ssh -o GatewayPorts=yes -L 0.0.0.0:2222:192.168.56.2:22 root@192.168.87.112
```

### 3. Настройки для стабильности соединения

В `~/.ssh/config` на клиенте:

```
Host 192.168.87.112
    ServerAliveInterval 60
    ServerAliveCountMax 3
    TCPKeepAlive yes
    Compression yes
```

### 4. Remote Port Forwarding (обратный туннель)

Если нужно открыть доступ с внешнего сервера к вашему локальному сервису:

```bash
ssh -R 8080:localhost:80 root@192.168.87.112
```

Теперь любой, кто подключится к роутеру на порт `8080`, попадёт на ваш локальный порт `80`.

### 5. SOCKS-прокси через Dynamic Port Forwarding

Для общего проксирования без привязки к конкретным портам:

```bash
ssh -D 1080 root@192.168.87.112
```

Настройте браузер на использование SOCKS5 прокси `127.0.0.1:1080` — и весь трафик пойдёт через роутер.

### 6. Работа с IPv6

`sshuttle` может работать с IPv6, если использовать соответствующие методы (`nft`, `tproxy`, `pf`):

```bash
sudo sshuttle -r root@192.168.87.112 ::/0
```

### 7. Остановка sshuttle

- **Ctrl+C** — корректное завершение с очисткой правил
- Если процесс завис: `sudo pkill -f sshuttle`

## Сравнение методов

| Метод | Сложность | Применение |
|-------|-----------|------------|
| Local Port Forwarding | Низкая | Для доступа к конкретным сервисам |
| sshuttle | Средняя | Для прозрачного доступа к целой подсети |
| ProxyCommand / Jumphost | Низкая | Для прямого SSH-доступа к внутренним хостам |
| SOCKS-прокси | Средняя | Для гибкого проксирования через браузер |

---

## Частые проблемы и их решение

**Ошибка:** `bind: Permission denied` при пробросе порта < 1024
**Решение:** Используйте `sudo` или выберите порт > 1024

**Ошибка:** `channel_setup_fwd_listener_tcpip: cannot listen to port`
**Решение:** Проверьте, что порт не занят: `sudo lsof -i :<порт>`

**Проблема:** Туннель рвётся через несколько минут бездействия
**Решение:** Добавьте `ServerAliveInterval 60` в настройки SSH

**Проблема:** `sshuttle` не видит внутренние хосты
**Решение:** Проверьте, что на роутере включен IP-форвардинг: `sudo sysctl -w net.ipv4.ip_forward=1`

**Проблема:** Браузер ругается на самоподписанный сертификат
**Решение:** Нажмите «Принять риск и продолжить» или добавьте сертификат в доверенные

---

## Заключение

Мы рассмотрели три основных способа организации доступа к внутренней сети через bastion-хост:

1. **Local Port Forwarding** — для доступа к конкретным сервисам
2. **sshuttle** — для полноценного VPN-подобного соединения
3. **ProxyCommand / Jumphost** — для прямого SSH-доступа к хостам

Выбор метода зависит от конкретной задачи: для единоразового доступа к сервису достаточно простого туннеля, для постоянной работы с целой сетью лучше подойдёт `sshuttle`.

> **Совет по безопасности:** Старайтесь не использовать `-L 0.0.0.0:...` без необходимости — это открывает доступ к туннелю с других машин в сети. Вместо этого используйте `127.0.0.1` и локальный SSH-клиент.

## Практический пример: Доступ к веб-интерфейсу по доменному имени через SSH-туннель

### Ситуация

Предположим, у вас есть удалённый сервер (`178.46.128.3`) с запущенным веб-приложением (например, Runtel), которое слушает порт `443` (HTTPS). Доступ к нему возможен только через SSH, а веб-интерфейс привязан к домену `lk-sirius.cprt.su`.

**Задача:** Открывать в браузере `https://lk-sirius.cprt.su/` так, чтобы запрос уходил через SSH-туннель на удалённый сервер.

---

### Решение

#### Шаг 1: Создаём SSH-туннель с пробросом порта

На **локальной машине** выполняем:

```bash
sudo ssh -L 443:127.0.0.1:443 root@178.46.128.3
```

**Почему нужен `sudo`:** Порт `443` является привилегированным (ниже 1024). Для его открытия на локальной машине требуются права суперпользователя.

**Расшифровка команды:**
- **`-L 443:127.0.0.1:443`** — локальный портфорвардинг: все соединения к порту `443` на вашем компьютере перенаправляются на порт `443` самого удалённого сервера (через его localhost)
- **`root@178.46.128.3`** — подключение к удалённому серверу от имени root

> **Важно:** Туннель работает **только пока активна SSH-сессия**. Не закрывайте окно терминала!

#### Шаг 2: Настраиваем локальный DNS (файл hosts)

Чтобы браузер направлял запросы к домену `lk-sirius.cprt.su` на ваш локальный компьютер (где висит туннель), добавляем запись в файл `/etc/hosts`:

**Linux / macOS:**
```bash
sudo nano /etc/hosts
```

**Windows:**
Запустите Блокнот от имени администратора и откройте:
```
C:\Windows\System32\drivers\etc\hosts
```

**Добавляем строку:**
```
127.0.0.1 lk-sirius.cprt.su
```

#### Шаг 3: Проверяем результат

Открываем браузер и переходим по адресу:
```
https://lk-sirius.cprt.su/
```

Браузер теперь обращается к `127.0.0.1`, где его ждёт SSH-туннель. Туннель перенаправляет запрос на удалённый сервер, и вы видите веб-интерфейс.

> **Примечание:** Браузер может ругаться на самоподписанный сертификат — нажмите «Принять риск и продолжить».

---

### Альтернативные варианты

#### Если порт 443 на локальной машине занят

Можно использовать другой локальный порт, например `8443`:

```bash
sudo ssh -L 8443:127.0.0.1:443 root@178.46.128.3
```

В этом случае в `/etc/hosts` также добавляем `127.0.0.1 lk-sirius.cprt.su`, а в браузере открываем:
```
https://lk-sirius.cprt.su:8443/
```

#### Если не хотим использовать sudo

Можно пробросить порт выше 1024 (например, `4443`) без прав root:

```bash
ssh -L 4443:127.0.0.1:443 root@178.46.128.3
```

Но тогда в браузере нужно будет указывать порт явно:
```
https://lk-sirius.cprt.su:4443/
```

---

### Почему это работает?

1. **Запись в `/etc/hosts`** говорит операционной системе: «Когда программа (браузер) хочет обратиться к `lk-sirius.cprt.su`, отправляй её на IP-адрес `127.0.0.1` (то есть на этот же компьютер)».
2. **SSH-туннель** слушает порт `443` на `127.0.0.1` и перенаправляет все входящие соединения на удалённый сервер.
3. **Удалённый сервер** принимает запрос как локальный (с `127.0.0.1`) и отдаёт веб-интерфейс.

Таким образом создаётся «мост» между вашим браузером и веб-сервером на удалённой машине, полностью зашифрованный через SSH.

---

### Проверка работоспособности

**На локальной машине (в другом окне терминала):**

Проверяем, что порт слушается:
```bash
sudo ss -tulpn | grep :443
```

Должны увидеть:
```
tcp   LISTEN 0      128    127.0.0.1:443    0.0.0.0:*
```

Проверяем резолвинг домена:
```bash
ping lk-sirius.cprt.su
```

Должен отвечать `127.0.0.1`.

**На удалённом сервере (в открытой SSH-сессии):**

Проверяем, что веб-сервер действительно слушает порт 443:
```bash
ss -tulpn | grep :443
```

Должны увидеть процесс (например, `runtel-iface` или `nginx`) на порту 443.

---

### Добавление в конфигурацию SSH (~/.ssh/config)

Чтобы каждый раз не вводить длинную команду, можно создать алиас в `~/.ssh/config`:

```bash
Host sirius-tunnel
    HostName 178.46.128.3
    User root
    LocalForward 443 127.0.0.1:443
    GatewayPorts no
```

Тогда для создания туннеля достаточно выполнить:
```bash
sudo ssh sirius-tunnel
```

---

### Важные замечания

1. **Безопасность:** Использование `root` на удалённом сервере — это потенциальная угроза. Лучше создать отдельного пользователя с ограниченными правами.
2. **Постоянство:** Для долгосрочной работы используйте `autossh`:
   ```bash
   sudo autossh -M 0 -o "ServerAliveInterval 60" -L 443:127.0.0.1:443 root@178.46.128.3
   ```
3. **Firewall:** Убедитесь, что на локальной машине открыт порт для туннеля (если используете `0.0.0.0` вместо `127.0.0.1`).

---

Этот пример отлично демонстрирует, как с помощью простого SSH-туннеля и локального hosts-файла можно получить доступ к веб-интерфейсу удалённого сервера так, будто он работает у вас локально.
